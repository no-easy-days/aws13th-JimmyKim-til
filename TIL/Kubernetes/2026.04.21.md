## 1. ConfigMap
- 클러스터에 존재하는 값(value)이나 설정을 참조하여 사용하고 싶을때 쓰는 오브젝트.
- `yaml` 파일 예시
  ```
  apiVersion: v1
  kind: ConfigMap
  metadata:
    name: info               # Deployment 또는 Pod가 이 이름을 통해 참조함
    namespace: default       # Namespace에 종속된다!! 주의!!
  
  data:                      # 여기부터 환경변수처럼 입력하면 된다. 
    db_host: 211.183.3.100
    username: root
  ```
- Pod 및 Deployment 참조 예시
  ```
  apiVersion: v1
  kind: Pod
  metadata:
    name: my-pod-env
  spec:
    containers:
    - name: my-container
      image: public.ecr.aws/docker/library/httpd:alpine
      env:
      - name: USERNAME
        valueFrom:              # 값을 어디서 받아오냐면,
          configMapKeyRef:      # configmap에서 참조해라.
            name: info          # 그 configmap의 이름은 info 이다.
            key: username
      - name: DATABASE
        valueFrom:
          configMapKeyRef:
            name: info
            key: db_host
  ```
- `data` 하위에 `|`(Pipe) 입력하면 여러 줄을 작성할 수 있다. ***대신, 텍스트만 가능!!***
  > 텍스트가 아닐 경우에는 이미지에 포함하는게 제일 깔끔하다. 말 그대로 config 관련 텍스트 정보 넘겨줄때만 쓴다.


## 2. Secret
- configMap과 동일하게 특정 변수, 설정을 컨테이너 안으로 주입하되, 보안수준이 다르다.
- 민감한정보(e.g. Database IP 주소, PW)를 외부에서 호출해서 쓰는 방식. base64 방식으로 인코딩이 되며, ***etcd 암호화***.
- 다른 오브젝트들은 api-server에 요청(get,describe)을 통해 정보를 조회할 수 있는 반면, Secret은 etcd에서 암호화로 조회 불가!
- `yaml` 파일 예시
  ```
  apiVersion: v1
  kind: Secret
  metadata:
    name: sec
  type: Opaque
  stringData:
    password: test123
  ```
  - ```Type: Opaque``` : `<Key> : <Value>` 쌍 형태로 정의하겠다는 것.
- `kubectl describe secret`을 해도, 실제 값은 조회가 불가능하다. Data가 몇 Byte인지 정도만 알려준다.

  
- Pod 및 Deployment 참조 예시
  ```
  apiVersion: v1
  kind: Pod
  metadata:
    name: pod-sec
  spec:
    containers:
    - name: sec-con
      image: public.ecr.aws/docker/library/nginx:alpine
      envFrom:                        #환경 변수를 가져올건데
      - secretRef:                    #secret 오브젝트에서 참조하겠다!!
          name: sec 
  ```
- 잘 참조했는지 결과는 ```kubectl exec```로 Pod에 들어가서 확인해야한다. (Pod 접근 권한 있어야 조회 가능)


## 3. volumeMounts 시 subPath 활용하기
-  configMap, secret에서 특정 Key, 특정 파일만 마운트 할때, MountPoint의 기존 내용은 유지하고 파일만 추가(수정)한다.
-  container의 파일을 호스트의 파일보다 우선하겠다
   > 원래는 호스트의 우선 순위가 높기때문에 기존의 컨테이너 내부의 파일은 삭제가 되는데 그걸 방지하겠다!


- `yaml` 파일 예시
  ```
  apiVersion: v1
  kind: Pod
  metadata:
    name: my-pod-conf
  spec:
    containers:
    - name: my-container
      image: public.ecr.aws/docker/library/nginx:alpine
      volumeMounts:
      - name: config-sample
        mountPath: /nginx.conf           # 마운트포인트.
        subPath: nginx.conf              # 파일만 마운트해라.
    volumes:
    - name: config-sample
      configMap:
        name: conf                 # 여기서 conf 라는 configMap은 이미 Pod로 띄워져 있는 상태이다.
        items:
        - key: nginx.conf
          path: nginx.conf         # 컨테이너 외부에 존재하는 파일. 내부 mountPath에 넣겠다.
  ```


## 4. StatefulSet
- 일반적으로 `replicas : `를 지정해서 Pod(Deployment)를 띄우면 그 Pod의 이름은 무작위로 생성되며, 추측이 불가능하다.
- `kind`를 `StatefulSet`으로 세팅하면 되고, Pod를 생성하면 ***"이름을 규칙적으로 생성 및 삭제***한다. 
- 각 Pod는 고유한 이름(예: `myapp-0`, `myapp-1`)과 함께 안정적인 네트워크 ID를 가지며, Pod가 재시작되도 동일하게 유지된다.
- **PersistentVolumeClaim**(PVC)을 사용하여 Pod마다 고유한 저장소를 제공하며, Pod가 삭제되더라도 데이터는 유지된다.
- ***데이터의 일관성과 안정성을 보장***해야 하는 경우 필수적이다!
    

  | **특징** | **StatefulSet** | **Deployment** |
  | --- | --- | --- |
  | Pod 이름 | 고유하고 안정적 (`pod-0`, `pod-1`) | 무작위 (`pod-abc123`) |
  | 순서 | 생성 및 삭제 순서 보장 | 순서 무관 |
  | 네트워크 ID | 안정적 (재시작 후에도 동일) | 무작위 (재시작 시 변경됨) |
  | 영구 스토리지 | 각 Pod마다 독립적인 PVC 사용 | 공유 볼륨 또는 임시 스토리지 사용 |
  | 사용 사례 | 데이터베이스, 캐시, 메시지 브로커 | 웹 서버, API 서버, 무상태 워크로드 |


- `yaml` 파일 예시
  ```
  apiVersion: apps/v1
  kind: StatefulSet         # 선언
  metadata:
    name: my-statefulset
  spec:
    serviceName: "my-service"          # 네트워크 ID를 위한 Headless Service를 정의
    replicas: 3                        # 생성할 Pod의 수. Pod의 이름은 my-service-<숫자> 로 붙는다
    selector:
      matchLabels:
        app: my-app
    template:
      metadata:
        labels:
          app: my-app
      spec:
        containers:
        - name: my-container
          image: ghcr.io/ej31/jeff-ex:1.3-amd64
          volumeMounts:
          - name: my-persistent-storage
            mountPath: /data
    volumeClaimTemplates:                   # 각 Pod에 독립적인 PVC를 생성
    - metadata:
        name: my-persistent-storage
      spec:
        accessModes: ["ReadWriteOnce"]
        resources:
          requests:
            storage: 1Gi 
  ```


## 5. DaemonSet
- `docker swarm`에서 `deploy.mode.global` 옵션과 비슷하다. ***각 Node마다 무조건 한개씩 Pod를 띄우고 싶을 경우*** 사용한다.
- `kube-system` Namespace의 `kube-proxy` 나 `metallb-system`의 `speaker`같은 리소스들이 이에 해당한다.
- ```kubectl get daemonsets.apps -n kube-system``` : 입력해보면 daemonSet이 활성화 된 것 확인 가능.
- `yaml` 파일은 `kind: DaemonSet` 만 지정하면 된다. 나머지는 일반적인 deployments 구성이랑 동일하게 하면 된다.


## 6. Container Health-Check
- `livenessProbe` 방식 : 실패 시 컨테이너를 재시작 - 컨테이너에 이상이 있다고 간주한다.
- `readinessProbe` 방식(많이 쓰임) : 실패해도 정상 동작하지만, Service에서 해당 Pod로 트래픽을 인가하지 않는다.
  (1) httpGet 방식
  - HTTP Method의 GET을 활용하여 Status Code를 확인하여 검증한다. (L7 Layer)
  ```
  apiVersion: v1
  kind: Pod
  metadata:
    name: nginx-pod
    labels:
      app: nginx
  spec:
    containers:
    - name: nginx
      image: 61.254.18.30:5000/nginx

    readinessProbe:                 # 선언 시작
      httpGet:                      # 방식 선언
        path: /                     # 타겟 디렉토리 (보통 웹루트 디렉토리 아니면 /health로 만들어둠)
        port: 80                    # 포트번호
      periodSeconds: 5              # Health-Check 주기
      initialDelaySeconds: 2        # Container 생성 완료 대기 시간
  ```
  - Pod(Deployments)는 정상 동작 하더라도, Health-Check가 문제 발생하면 ***Service가 Pod로 트래픽을 보내지 않음.***
    - `kubectl describe pods` 확인 시 정상이나, `kubectl describe service` 확인 시 ***"EndPoint 공백 처리"***.


  (2) tcpSocket 방식
  - http 프로토콜이 아닌 tcp 프로토콜을 통한 소켓 통신 정상 여부로 검증한다. (L4 Layer)
  ```
  apiVersion: v1
  kind: Pod
  metadata:
    name: redis
    labels:
      app: redis
  spec:
    containers:
    - name: redis
      image: 61.254.18.30:5000/redis

    readinessProbe:
      tcpSocket:
        port: 6379                        # 해당 포트로 Socket 통신 정상 여부를 확인한다.
      initialDelaySeconds: 5
      periodSeconds: 3
  ```

  (3) Command 방식
  - 특정 명령어가 잘 수행되는지를 바탕으로 검증한다.
  ```
  apiVersion: v1
  kind: Pod
  metadata:
    name: mysql
  spec:
    containers:
    - name: mysql
      image: 61.254.18.30:5000/mysql
      env:
      - name: MYSQL_ROOT_PASSWORD
        value: "password"

    readinessProbe:
      exec:
        command: ["mysqladmin","ping","-ppassword"]     # 이 명령의 리턴 값이 잘 받아오는지 확
      initialDelaySeconds: 5
      periodSeconds: 3
  ```
