## 1. LoadBalancer
- AWS의 NLB와 유사한 개념. ***"클러스터 관리자의 도움이 필요"*** 한 서비스 타입. (L4 계층)
- AWS의 EKS 같은 쿠버네티스 클러스터의 경우, 클라우드 서비스 관리자 및 제공자(AWS)가 LB를 제공해줄 수 있다.
- 하지만, 온프레미스에 구성한 클러스터는 그렇지 않기 때문에, 누군가가 LB를 생성하면서 노드 대역대의 IP를 할당해줘야한다.
- 그 누군가는 ***MetalLB를 설치***하면 된다.
  > LB는 로드밸런서, Metal = Bare-metal. 내 온프레미스 클러스터에 로드밸런서 기능을 추가하는 오브젝트 (Add-On)


- 설치 방법
  - ```wget https://raw.githubusercontent.com/metallb/metallb/v0.15.3/config/manifests/metallb-native.yaml``` : Manifest 파일 다운로드
  - ```kubectl apply -f metallb-native.yaml``` : Manifest 파일로 실행

- ```kubectl get pod -n metallb-system``` 입력 해보면, ```speaker``` 로 명시된 Pod를 볼 수 있다.
- Speaker : LB에 부여될 IP를 외부에 알려주는 역할 + 중복 방지 (나를 알려야 찾아올 수 있으니까)   
- ```vi config-metal.yml``` : LB가 생성될 때 뿌려줄 IP 범위 설정
  ```
  apiVersion: metallb.io/v1beta1
  kind: IPAddressPool
  metadata:
    name: first-pool
    namespace: metallb-system
  spec:
    addresses:
    - 211.183.3.200-211.183.3.240 # 안겹치게 수정. LB가 부여받을IP범위
  ---
  apiVersion: metallb.io/v1beta1
  kind: L2Advertisement
  metadata:
    name: example
    namespace: metallb-system
  ```
- ***"MetalLB"*** 구성이 완료되면, 이제 배포할 svc의 타입을 ```LoadBalancer``` 로 지정하면 IP를 부여받는다.


## 2. Ingress
- AWS의 ALB와 유사한 개념. ***"PATH"*** 기반 라우팅을 위한 오브젝트 (L7 계층)
  > /board 로 가면 board앱으로, /login으로 가면 login 앱으로 간다는 의미


- #### 일반적인 svc는 PATH 기반 라우팅이 불가능하다. 한 종류의 label만 품을 수 있기 때문이다.
- 여러 개의 컨테이너에 각 기능들을 구현하면 PATH로 라우팅이 가능해야 하는데, 이때 ***'Ingress'*** 라는 리소스가 필요하다.


  <img width="1057" height="550" alt="image" src="https://github.com/user-attachments/assets/80bec919-dc3c-4d53-874d-95fcf9e5e5a3" />

  
- Ingress를 구성하려면 ```Ingress Controller``` 가 필요하다. (AWS EKS에서는 ```LoadBalancer Controller```로 칭함)
- 설치 방법
  - https://kubernetes.github.io/ingress-nginx/deploy/#bare-metal-clusters : Manifest 파일 wget 진행
  - ```vi deploy.yaml``` : 365번째 라인 ```type: NodePort``` 를 ```type: LoadBalancer```로 수정
  - ```kubectl apply -f deploy.yaml```: Ingress Controller 활성화
  - ```kubectl get services -n ingress-nginx``` : 활성화 완료 시 ***EXTERNAL-IP*** 정보 확인 가능!

- 설치 완료 후, ingress controller 사용을 위한 yaml 파일
  ```
  apiVersion: networking.k8s.io/v1
  kind: Ingress
  metadata:
    name: ing-ip
    annotations:          # "labels"와 비슷한지만 주로 부가적인 기능 명시. EKS 연동하면 자주 사용함!!!
      nginx.ingress.kubernetes.io/rewrite-target: /    #/httpd로 접속하면 / 경로를 바꿔주겠다.
  spec:
    ingressClassName: "nginx"    # 종류는 많지만 nginx를 선택한다.
    rules:
      - host: rapa.com      # 호스팅할 주소. IP는 불가능함!
        http:
          paths:
          - path: /httpd
            pathType: Prefix
            backend:
              service:
                name: svc-ipnginx     #서비스의 이름 매핑
                port:
                  number: 80     #서비스의 포트
   ```

  
- ### 서비스를 마치 Domain 주소처럼 활용한다. 느낌을 이해하자.
- ```kubectl describe ingress <ingress이름>``` : ***EndPoint Target이 잘 매핑되었는지 확인***하자.
- ```host``` 값에 주소를 입력하기 때문에, ***별도의 DNS Server를 통해서 EXTERNAL-IP와 매핑***시켜줘야 한다!!!


## 3. PV(Persistent Volume), PVC(Persistent Volume Claim)
- **PV** : 클러스터에서 관리되는 실제(물리적 또는 가상) 스토리지 리소스
- **PVC** : 어플리케이션이 필요한 스토리지 요구사항(용량, 접근 모드 등)을 기술해 PV 할당 요청(Request) 리소스
  > Pod는 PVC를 마운트함으로써 **영구(Persistent) 스토리지**를 활용할 수 있게 된다!


<img width="1576" height="1028" alt="image" src="https://github.com/user-attachments/assets/627a1961-fb74-440d-9326-387330192aff" />


- **PV 생성**: “클러스터에 이만큼의 스토리지 자원을 등록한다.”
- **PVC 생성**: “이런 용량·접근 모드로 스토리지를 쓰겠다.”
- **바인딩(Bound)**: "PVC 요구사항을 충족하는 PV가 있으면 자동으로 연결시켜라!"
- **데이터 영속성**: Pod가 죽어도 PV는 남아있으니까 PVC가 PV를 유지(또는 새로운 Pod가 동일 PVC를 재마운트)할 수 있다.

- ***```pv-sample.yaml``` 예시***
  ```
  apiVersion: v1
  kind: PersistentVolume
  metadata:
    name: nfs-pv1
  spec:
    capacity:
      storage: 1Gi     # 할당할 PV의 맥시멈 용랑
    accessModes:       
      - ReadWriteMany      # 스토리지 엑세스 모드
    persistentVolumeReclaimPolicy: Retain     #Reclaim 방식
    nfs:
      path: /shared/pv1
      server: 211.183.3.100
  ```


- ***```pvc-sample.yaml``` 예시***
  ```
  apiVersion: v1
  kind: PersistentVolumeClaim
  metadata:
    name: nfs-pvc1
  spec:
    accessModes:
      - ReadWriteMany
    resources:
      requests:
        storage: 1Gi
  ```


- #### accessModes 및 Reclaim Policy는 pv와 pvc를 일치시켜줘야 한다!
  - persistentVolumeReclaimPolicy
    - **Retain** : PVC 삭제되도 PV는 유지하겠다. / **Delete** : PVC 삭제되면 PV도 삭제하겠다.
  - accessMode
    - **ReadWriteOnce(RWO), ReadOnlyMany(ROX), ReadWriteMany(RWX), ReadWriteOncePod(RWOP)**
- #### pvc의 용량은 pv보다 같거나 작아야 하며, pv를 선택해서 bind하는 것이 아닌 pv 중에서 선택되는 구조.
- PV는 Static / Dynamic Provisioning 방식이 있다. 

- Deployment 및 Pod 사용 예시
  ```
  apiVersion: v1
  kind: Pod
  metadata:
    name: nfs-pod1
  spec:
    containers:
    - name: nfs-container
      image: 61.254.18.30:5000/ipnginx
      volumeMounts:
      - name: nfs-volume   # 매핑
        mountPath: /remote
  volumes:
  - name: nfs-volume        # 매핑
    persistentVolumeClaim:
      claimName: nfs-pvc1   # pvc name 매핑
  ```


## 4. RBAC(Role Based Access Control)
- ***"Role", "RoleBinding", "ServiceAccount" 3개의 리소스로 구성***된다.


  <img width="835" height="555" alt="image" src="https://github.com/user-attachments/assets/1dacffbb-2d5b-4fb5-8489-f748c26a40b3" />


- `ServiceAccount` ≒ AWS Role
- `Role` ≒ AWS Policy
  - 특정 Namespace에 대한 권한이 할당된다.
- 생성된 `Role`을 `ServiceAccount`를 부여하는 작업이 `RoleBinding`이다.
- `ClusterRole`
  - 클러스터 전체(모든 Namespace)에 대한 권한이 할당된다.
- 생성된 `ClusterRole`을 `ServiceAccount`에 부여하는 작업이 `ClusterRoleBinding`이다.  

  
## 5. StorageClass를 활용한 Dynamic Provisioner 구현
- ***PVC Request 발생 시 자동으로 PV를 생성***한다. 디렉토리를 미리 생성하거나, 수동으로 생성할 필요가 없다.
- AWS EKS에서는 `efs csi Driver`를 통해 이 기능을 구현할 수 있다.

- 구현 방법
  - 우선, PV 및 PVC ```yaml``` 파일 작성 시 ```spec``` 하위에 ```storageClassName: ``` 으로 작성한다.
  - ```git clone https://github.com/kubernetes-sigs/nfs-subdir-external-provisioner``` : 관련 파일 다운로드
    - ```rbac.yaml``` : SA 및 Role 할당 구성. ***PV를 생성할 수 있는 SA를 생성***한다.
    - ```class.yaml``` : 스토리지 클래스 구성. ```deployment.yaml```에 선언된 ```PROVISIONER_NAME```과 매핑 필요하다!
    - ```deployment.yaml``` : 다이나믹 프로비저너 실제 구성을 ***Deployment(Pod)에 부여***한다.
    - ```test-claim.yaml``` : 테스트용 PVC 구성.
