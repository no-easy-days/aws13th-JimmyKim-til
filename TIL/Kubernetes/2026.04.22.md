## 1. HPA(Horizontal Pod Auto-Scaler)
- 말그대로, 트래픽 처리량에 따라 ***Pod의 "갯수"*** 를 자동으로 조절한다.


  <img width="1066" height="665" alt="image" src="https://github.com/user-attachments/assets/7324fab8-7fab-44e5-bfb3-2c732e832eba" />


  > HPA는 이름에서 처럼 Scale-Out & Scale-In 방식이다. Container가 구현된 이미지의 Spec은 관계없다.


- 지표를 분석해야 하기 때문에, 대상 Pod를 지속 모니터링하고 있는 `Metric Server`가 있어야 한다.
- Metric Server 설치 방법
  - `curl -Ls https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml -o metric.yml` : Manifest 다운로드


    <img width="1102" height="303" alt="image" src="https://github.com/user-attachments/assets/aa6ba0ba-7c71-425c-b37a-57bb0a18392c" />


  - `kubectl apply -f metric.yml` : Metric Server 역할을 하는 Pod 실행


- 모니터링 대상이 될 Service(Deployment 포함) `yaml` 파일 예시
  ```
  apiVersion: v1
  kind: Service
  metadata:
    name: my-svc
  spec:
    selector:
      app: my-web
    ports:
    - port: 80
      targetPort: 80
  ---
  apiVersion: apps/v1
  kind: Deployment 
  metadata:
    name: my-dep
  spec:
    replicas: 3
    selector:
      matchLabels:
        app: my-web
    template:
      metadata:
        labels:
          app: my-web
      spec:
        containers:
        - image: 61.254.18.30:5000/ipnginx
          name: my-web-con                  # 여기까진 Service ~ Deployment로 이어지는 일반적인 Manifest 
          resources:                        # Metric을 제공해야 하므로, 오브젝트 생성 시 요청할 자원을 선언한다.
            requests:                       # 생성 시 요청할 CPU 사용량, 메모리 용량
              cpu: "10m"
              memory: "64Mi" 
            limits:                         # 할당할 최대 CPU 사용량, 메모리 용량
              cpu: "20m"
              memory: "128Mi"
  ```

- HPA `yaml` 파일 예시
  ```
  apiVersion: autoscaling/v1
  kind: HorizontalPodAutoscaler                    # 여기서 HPA로 선택한다.
  metadata:
    name: hpa-nginx
  spec:
    maxReplicas: 5                                 # 최대 복제본 갯수
    minReplicas: 1                                 # 최소 복제본 갯수
    scaleTargetRef:                                # 타겟이 되는 걸 지정할텐데
      apiVersion: apps/v1                     
      kind: Deployment                             
      name: my-dep                                 # Deployment의 이름을 매핑한다!!
    targetCPUUtilizationPercentage: 50             # CPU 사용량 임계치를 50%로 한다.
  ```
  - `Deployment`의 Manifest에 `replicas: ` 값이 들어가있더라도, `HPA` Manifest의 min~max Replica값을 자동으로 따라간다!
    > 처음에 복제본 만들어 놓더라도 트래픽 없으면 min으로 줄어듬.


## 2. HAproxy
- 3개의 Master Node가 있을 경우, ***1개의 Node만 Active, 2개는 Standby 상태로 구성***한다.
- Active Node가 문제 생기면 Standby Node가 Fail-Over되오 Active 상태로 전환된다.
- #### 새로운 Master Node를 선출하기 위해서는 Master Node의 갯수를 홀수개(3,5,7,9....)로 가져가야 한다!


  <img width="1122" height="598" alt="image" src="https://github.com/user-attachments/assets/0ae7aced-9202-4010-9df0-7597b60bae4c" />


  (1) HAproxy Server 구성
    - ```apt update -y && apt install -y haproxy``` : HAproxy 패키지 설치
    - ```vi /etc/haproxy/haproxy.cfg``` : HAproxy 컨피그 파일 편집

        ```
        frontend kubernetes-master-lb
        bind 0.0.0.0:6443
        option tcplog
        mode tcp

        default_backend kubernetes-master-nodes

        backend kubernetes-master-nodes
        mode tcp
        balance roundrobin
        option tcp-check
        option tcplog
        server k8s-master1 <첫번째 Master Node의 IP주소>:6443 check
        server k8s-master2 <두번째 Master Node의 IP주소>:6443 check
        server k8s-master3 <첫번째 Master Node의 IP주소>:6443 check

        ※ 이 예시에서는 Master Node가 3대인 상태로 구성했다. 갯수 변경 시 맞춰서 구성하며 뇐다.
        ```
    - ```systemctl enable --now haproxy``` : HAproxy 데몬 활성화
