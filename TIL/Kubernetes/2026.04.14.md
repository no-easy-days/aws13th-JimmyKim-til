## 1. K8s 좀 더 쉽게 사용하기
- ```apt-get install -y bash-completion``` : bash 터미널 명령어 입력 자동화 서포팅 패키지 설치 
- ```echo 'source <(kubectl completion bash)' >>~/.bashrc``` : kubectl 명령어 터미널에 자동화 세팅
- ```echo 'source <(kubeadm completion bash)' >>~/.bashrc``` : kubeadm 명령어 터미널에 자동화 세팅
- ```kubectl completion bash >/etc/bash_completion.d/kubectl``` : k8s 명령어 할당
  > 이제 명령어 입력시 "k"만 쳐도 자동으로 단축 명령이 입력된다.


## 2. "label" 이란?
- ***K8s의 다양한 리소스를 제어(Desired State 및 Current State 확인 등)하고 찾아가기 위해*** 말 그대로 라벨을 붙이는 것. 


   ```
   ---
   apiVersion: v1
   kind: Pod
   metadata:
     name: test-label-pod

     labels:
      app: my-web

   spec:
     containers:
     - image: public.ecr.aws/docker/library/httpd:alpine
       name: test-label-con
   ```


- ```kubectl describe pod test-label-pod``` : ```test-label-pod``` 이름의 Pod 조회하면 ```Labels:    app=my-web``` 형태로 확인할 수 있음!


## 3. "ReplicaSet" 
- Pod : 대표적인 K8s의 리소스. 컨테이너 이미지를 띄우기 위한 ***"최소 배포 단위"***.
- ReplicaSet은 ```spec``` 하위에 선언하며, Pod보다 한단계 상위개념이다. 말 그대로 ***Pod의 복제본***을 생성한다.
- 버전관리에 반드시 필요한 리소스이며, 스케일링이 가능하다. ex) Pod갯수 2개 → 4개 등으로 변경
- ```selector``` 를 통해서 복제할 Pod의 ***"label"을 선택하여 복제한다!*** 


  <img width="1017" height="568" alt="image" src="https://github.com/user-attachments/assets/520777db-19d7-49e2-9d61-c57fcd13247a" />


  ```
  apiVersion: apps/v1
  kind: ReplicaSet
  # Pod로 선언하지 않고 "ReplicaSet"으로 선언했음!
  
  metadata:
  name: test-rep

  spec:
    replicas: 2
    selector:
    # label 기반으로 복제 대상을 선택한다 라는 뜻

      matchLabels:
      # 복제본을 만들 label 매핑
        app: my-web

    template:
    # Pod의 Manifest 정보를 담아 템플릿 형태로 선언
  
      metadata:
        labels:
          app: my-web
          # 위에 matchLabels에 선언되어 있는 걸 매핑해서 할당해야 에러가 나지않는다! 

      spec:
        containers:
        - image: public.ecr.aws/docker/library/httpd:alpine
          name: test-pod-con

   ※ yaml 파일 패턴을 기억하자!
   ```


- ReplicaSet의 일부분으로 구성하고 있는 Pod을 강제로 삭제할 경우, K8s의 ReplicaSet 컨트롤러가 지속 모니터링하여 자동 재생성한다.


## 4. Deployment
- 말 그대로 배포를 하기 위한 리소스


  <img width="1043" height="702" alt="image" src="https://github.com/user-attachments/assets/05f6565a-e00b-4ae2-9d0a-4c9bd8826ffe" />


  ```
  apiVersion: apps/v1
  kind: Deployment
  # 위에 ReplicaSet을 생성한 yaml파일과 다른건 여기 뿐이다!
   
  metadata:
  name: test-rep
  spec:
    replicas: 2
    selector:
      matchLabels:
        app: my-web

    template:  
      metadata:
        labels:
          app: my-web

      spec:
        containers:
        - image: public.ecr.aws/docker/library/httpd:alpine
          name: test-pod-con
   ```  

- Deployment를 만들면 replicaset과 pod 자동 생성된다.
  > Deployment는 ReplicaSet을 품고있고, ReplicaSet은 Pod를 품고있기 때문이다!


## 5. Namespace
- 위에서 설명한 다양한 분류를 갖는 리소스들을 공간적으로 분리, 격리하기위한 리소스.
  > 별도로 명시하지 않으면 Namespace가 "default" 라는 이름으로 자동 세팅된다! 


- Namespace 별로 접근 제어 및 권한 부여가 가능하다.
- 명령어
  - ```kubectl create ns <Namespace이름>``` : Namespace 선언 및 생성
  - ```kubectl apply -f <Manifest파일> -n <Namespace이름>``` : 선언한 Namespace에 "deploy" 생성
  - ```kubectl get pod --all-namespaces``` : 모든 Namespace에 존재하는 Pod 정보 확인
- yaml 파일로 Namespace 정의
  ```
  apiVersion: v1
  kind: Namespace
  # 리소스 종류에 이렇게 할당하면 됨.

  metadata:
    name: test-namespace

  .
  .
  .

  ```
