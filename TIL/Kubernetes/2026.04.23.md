## 1. NodeLabel
- Node 단위에 **키-값(key-value) 쌍** 형태로 붙여서 Metadata 정보를 설정할 수 있음.
- 배포 후에도 `kubectl` 명령을 통해 언제든 라벨을 추가/수정/삭제할 수 있다. **"동적 변경 가능"**
- `kubectl get nodes --show-labels` : 노드에 설정된 라벨 정보 조회
- `kubectl label node [노드명] [Key=Value]` : Node에 Label 생성 및 지정(예: gpu=false)
- `kubectl label nodes [노드명] [Key=Value] --overwrite` : 기존의 Label 덮어쓰기
- `kubectl label nodes [노드명] [Key]-` : Label 삭제
  
- 대표적인 활용 가능한 상황
  - **환경별 Node 구분**
    - 개발 환경용(`env=dev`)과 운영 환경용(`env=prod`) 노드를 분리하고 싶을 때.
  - **리소스 특성별 Node 구분**
    - CPU, RAM 등 자원이 큰 노드를 `size=large`, 작은 노드를 `size=small` 등으로 태그하여 Pod 배치 분산 용이
  - **특수 하드웨어 구분**
    - GPU 장착 노드를 `accelerator=gpu` 등으로 라벨링 하여, GPU가 필요한 워크로드만 해당 노드로 보낼 때
  - **네트워크/보안 정책 구분**
    - 민감 데이터가 필요한 Pod는 보안이 철저한 노드(`secure=true`)에만 배치하도록 구성

- Pod 배포 시 `nodeSelector`, `NodeAffinity` 등을 통해 ***특정 라벨이 있는 Node에만 스케줄링***하도록 설정한다.


## 2. NodeSelector
- K8s에서 ***"스케줄링"을 단순화***하기 위해 도입된 메커니즘
- Pod가 `<key>:<value>` 형태의 `NodeLabel`을 갖추고 있는 Node에서만 동작하도록 필터링한다.
- Pod(Deployment)의 Manifest 파일 內 `spec: ` 하위 항목에 `nodeSelector: { key: value }`를 추가하여 label을 붙여준다.
- 오직 ***`NodeLabel`과의 “정확한 일치”만 조건***으로 사용 가능하다! AND,OR 등은 구현 불가능하다.
- 대표적인 활용 가능한 상황
  - **특정 노드에만 설치되어 있는 하드웨어를 사용해야 하는 경우** (예: GPU, SSD, 전용 NIC 등)
  - **워크로드 테스트**를 위해 Node를 구분하고 싶을 때 (예: 개발용, QA용, 프로덕션용 등)
  - **원초적으로 단순화된 스케줄링**이 필요한 환경 (예: 소규모 클러스터에서 ‘이 노드에만 올려야 해!’처럼 명시가 필요한 경우)


## 3. NodeAffinity
-  Pod가 특정 노드의 `NodeLabel`이나 속성을 기준으로 배치되는 방식을 세밀하게 조정하여 유연성을 챙긴 메커니즘
  > Affinity = 친밀도, 인력


- **"`NodeLabel`이 달린 노드에 스케줄링되었으면 좋겠다"** 하는 의미.
- Manifest `yaml` 파일을 바탕으로, API Server가 `spec: ` 정보를 `etcd`에 저장한다.
- 그 이후, Scheduler가 스케줄링 과정에서 Pod의 NodeAffinity 설정을 확인하고 조건에 맞는 Node를 찾아서 Pod를 배치한다.
- NodeAffinity는 `NodeLabel`이 있는 노드에 ***우선적으로 또는 반드시 배치하도록 설정***할 수 있다.


### ※ 중요 규칙


(1) `requiredDuringSchedulingIgnoredDuringExecution` : "필요조건"
  - Pod를 스케줄링할 때(`DuringScheduling`) 해당 조건이 “반드시(`required`)” 만족되는 노드에만 스케줄링한다.
  - 그런데, Pod가 실행중(`Execution`)인 동안에는 무시(`Ignored`)된다.
    > 쉽게 말해서 K8s의 Scheduler가 Pod를 처음 스케줄링할 때 특정 조건을 반드시 충족해야 하고, 다른 애를 쫓아내진 않는다.
  
  
(2) `preferredDuringSchedulingIgnoredDuringExecution` : "선호조건"
  - Pod를 스케줄링할 때(`DuringScheduling`) 해당 조건이 “가능하면(`preferred`)” 충족하는 노드에 스케줄링한다.
  - 그런데, Pod가 실행중(`Execution`)인 동안에는 무시(`Ignored`)된다.
    > 쉽게 말해서 K8s의 Scheduler가 Pod를 처음 스케줄링할 때 특정 조건(노드 라벨 등)을 선호하지만, 꼭 지킬 필요는 없다.


- 대표적인 활용 가능한 상황
  - **특정 하드웨어 스펙 노드에만 배치하고 싶을 때**
    - GPU 달린 노드에서만 실행되어야 하는 ML/딥러닝, DB 워크로드
  - **지역성(Region/Zone)에 따라 배포하고 싶을 때**
    - AWS의 여러 AZ에 분산 배치해서 고가용성 고려
  - **특정 노드와 충돌을 피하고 싶을 때**
    - 리소스가 제한된 노드에는 중요 워크로드를 배치하지 않으려고 할 때


- Node Seletor와 마찬가지로, 각 Worker Node에 `kubectl label nodes` 명령을 통해 라벨을 붙여줘야 한다.
- `yaml` 파일 예시
  ```
  apiVersion: apps/v1
  kind: Deployment
  metadata:
    name: my-aff
  spec:
    replicas: 4
    selector:
      matchLabels:
        app: my-httpd
    template:
      metadata:
        labels:
          app: my-httpd
      spec:
        containers: 
          - name: httpd-pod-con
            image: 61.254.18.30:5000/ipnginx
  
        affinity:                    # containers: 와 동등한 레벨에서 작성 시작!!
          nodeAffinity:

            ※ preferred 예시
            preferredDuringSchedulingIgnoredDuringExecution:
              - weight: 1
                preference:          # perference이기 때문에 하나 더 조건을 부여.  
                  matchExpressions:
                    - key: gpu
                      operator: In
                      values:
                        - 'false'

            ※ required 예시
            requiredDuringSchedulingIgnoredDuringExecution:
              nodeSelectorTerms:     # 아래 조건을 만족하는 노드에만 스케줄링한다. (Selector 기법 차용)
              - matchExpressions:
                - key: gpu
                  operator: In
                  values:
                  - "false"
  ```
- `weight` 점수를 다르게 부여하여 우선 순위를 매길 수 있다.
- 일반적인 Pod보단 ***Deployment, StatefulSet과 함께 적용***하여 실무에서 많이 쓰인다.


## 4. Nodename
- `nodeName` : 스케줄러가 관여하지 않고(스킵하게 만들고), 곧바로 ***"지정된" 노드***에 Pod를 올리는 방법.
  - K8s 스케줄러는 “내가 Scheduling 하지 않아도 되겠다” 하고 스케줄링을 생략 & 지정된 노드에 Pod 생성됨.
  - 노드이름을 하드코딩하게 되므로 장애 발생 시 자동으로 다른 노드에 옮겨지지 않는다(수동 복구 필요!).
  - `NodeSelector`나 `NodeAffinity` 조건보다 ***"우선적"으로 효력을 발휘***한다.
- `yaml` 파일 예시
  ```
  apiVersion: v1
  kind: Pod
  metadata:
    name: nodename-demo
  spec:
    nodeName: worker3
    containers:
    - name: nodename-demo-container
      image: ghcr.io/ej31/jeff-ex:1.3-amd64
      ports:
        - containerPort: 3000
  ```
- `Pod` Spec에서만 사용할 수 있으며, `Deployment`나 `DaemonSet` 같은 상위 수준 컨트롤러에서는 설정 불가!!
- 대표적인 활용 가능한 상황
  - **특정 자원이 필요한 노드**에서만 실행
    - GPU, 큰 메모리가 필요한 경우 등, 해당 노드에서만 사용하는 Pod를 강제 배치해야 할 때
  - **고정된 개발/테스트 환경**
    - 정확히 특정 노드에서 개발 환경을 재현해야 하는 경우(네트워크 IP가 특정 노드에만 열려 있다거나)
  - **Node Affinity** 등 다른 스케줄링 방법이 복잡할 때
    - 별도의 라벨링 없이, 간단히 “이 노드에만 실행할래!” 할 때
  > GPU 기반 ML 노드나 NoSQL, 캐싱 서버 등 특정 네트워크/스토리지 자원이 필요한 상황 등


## 5. Taint & Tolerations  
- `taint` : 오염, 노드에 “오염” 조건을 걸어 지정된 Pod만 스케줄링하도록 제한한다.
- `tolerate` : 허용하다, 노드가 가진 “오염(`Taint`)”된 조건을 Pod가 “버틸 수 있게” 하는 일종의 ***"허용 조건"***.
- Pod가 스케줄링될 때 적용된다.

  
- 대표적인 활용 가능한 상황
  - **전용 리소스 노드**: GPU나 고성능 SSD를 가진 노드에 특정 Pod만 배포하고 싶을 때
    - GPU 노드는 `gpu=true:NoSchedule` 같은 taint를 달고, GPU가 필요한 Pod는 “toleration”을 추가하여 배포
  - **테스트/개발 구분**: 특정 노드는 QA 용도이고, 다른 노드는 프로덕션 용도인 경우
    - `env=dev:NoSchedule` taint를 통해 해당 노드에는 `dev` Pod만 가도록 설정
  - **보안 및 정책 준수**: 민감 데이터 처리를 위해, 보안이 강화된 노드로만 제한적으로 Pod를 배포


- `kubectl taint nodes [노드명] [키=값:효과]` : Taint를 특정 Worker Node에 추가한다.
  - `NoSchedule` : Pod를 스케줄링하지 않겠다는 의미, 쉽게 말해서 Pod 생성 안하겠다.
  - `PreferNoSchedule` : 스케줄러가 다른 노드를 **"우선적"으로 선택**하지만, 여의치 않으면 할당된다.
  - `NoExecute` : 새 Pod도 허용하고 기존 Pod도 유지한다. (없으면 기존 Pod 축출하고 새로운 Pod도 거부)


- `yaml` 파일 예시
  ```
  apiVersion: apps/v1
  kind: Deployment
  metadata:
    name: deployment-with-toleration
  spec:
    replicas: 1
    selector:
      matchLabels:
        app: toleration-app
    template:
      metadata:
        labels:
          app: toleration-app
      spec:
        tolerations:                      # toleration 선언, spec 하위에 항목으로 집어 넣는다.
          - key: "env"                    # 노드에 설정된 taint의 Key 값
            operator: "Equal"             # 비교 연산자 지정. (Equal / NotEqual / DoesNotExist, Exists)
            value: "dev"                  # 노드에 설정된 taint의 Value 값
            effect: "NoSchedule"          # 적용 효과. (NoSchedule, PreferNoSchedule, NoExecute)
        containers:
          - name: toleration-container
            image: ghcr.io/ej31/jeff-ex:1.3-amd64
            ports:
              - containerPort: 3000
  ```
