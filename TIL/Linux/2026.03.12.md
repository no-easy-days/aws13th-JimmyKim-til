## 📝 배운 내용 (Today I Learned)

### 가상화 개요
- Server 가상화(Hypervisor) / Container 가상화(Docker, K8s)
- Server 가상화는 Hypervisor 구축 위치에 따른 분류 / 명령 전달 방법에 따른 분류가 있다.


1. Hypervisor 구조에 따른 분류
   <img width="832" height="460" alt="image" src="https://github.com/user-attachments/assets/a7ef6ec1-1662-43c5-94cf-5a7ff47ecb2a" />


   ※ 가상 머신을 ***"Domain"*** 이라고 칭한다. 헷갈리지 말자.


   - type 1 : Host OS와 Hypervisor가 통합되어 구축된다. 성능 & 자원 효율이 높다. "Bare-Metal Virtualization"이라고도 부름
     - 대표 예시 : VMware ESXi, Hyper-V, Xen, KVM
   - type 2 : Host OS "위에" Hypervisor가 구축된다. 자원 효율 낮고, Overhead(손실) 다소 발생하지만 Host에서 별도 작업 가능.
     - 대표 예시 : VMware Workstation, Oracle VirtualBox, 


2. 명령 전달 방법에 따른 분류
   <img width="867" height="428" alt="image" src="https://github.com/user-attachments/assets/42c04a51-2447-40bf-9902-530e53e43e2a" />


   - 전가상화 (Full)
     - domain 0(최초 구축된 가상 머신)가 나머지 domain을 관장한다.
     - guest VM에서 발생하는 모든 명령에 Hypervisor가 개입한다.
     - guest VM은 스스로가 가상 머신이라는 것을 인지하지 못한다. ***guest OS를 건들 필요가 없다.*** 바꾸려면 Host OS 수정한다.
     - OS 호환성이 높다.
       - 대표 기술 : Intel-VT-x, AMD-V (VMware Workstation에서 전가상화를 구현 실습할 때, 얘네를 활성화하는 옵션을 켜준다!!)
       - 대표 플랫폼 : ESXi, KVM, Hyper-V
   - 반가상화 (Para)
     - guest가 VM에서 명령어를 쓸때 "Hypervisor를 통해 전달할 명령 = ***HyperCall***" / "그렇지 않은 명령" 을 구분한다.
     - guest VM은 스스로가 가상머신이라는 것을 인지한다. ***guest OS를 일부 수정해야한다.*** → CPU 제어 명령을 다르게 호출.
     - OS 호환성이 전가상화에 비해 상대적으로 낮다.


### KVM 구축
- Kernel-based Virtual Machine. ***리눅스 Kernel 자체가 Hypervisor 처럼*** 구동된다.
- **Host OS**에 "br0" 라는 ***"가상 브릿지 인터페이스"를 추가하여, Guest OS를 Host와 같은 대역대에 있도록*** 만든다.


1. KVM 구축을 위한 패키지 설치
   ```
   yum install -y qemu-kvm libvirt virt-install virt-manager virt-viewer
   ``` 


2. qemu 설정값 변경을 통한 입력 활성화
   - vi /etc/libvirt/qemu.conf
     > set nu 입력해서 442번줄 or "root"를 검색해서, user="root" 랑 group="root" 주석 해제. 권한을 부여하는 행위


3. GUI 환경에서 Virtual Machine Manager 접속 및 사용
   - iso 파일 및 Guest VM의 볼륨 저장 위치 : /var/lib/libvert/images/


#### KVM은 Type 1 가상화인가??
<img width="820" height="475" alt="image" src="https://github.com/user-attachments/assets/29f45c66-0f70-4703-ada5-0d29becc1e77" />


- kernel = 하드웨어를 제어하기위한 다양한 모듈들의 집합
- Virt-Manager가 마치 Hypervisor처럼 보이지만, KVM을 관리하기 위한 다양한 도구중에 하나일뿐이다.
- ***kvm = CPU와 RAM을 가상화(이것이 핵심, 근데 이건 이미 Linux 커널 자체에 설치되있다!)***
- qemu = cpu, ram, 다양한 주변장치(스토리지,네트워크,키보드 등..)를 생성(에뮬레이션) 및 관리
- libvirt = qemu-kvm을 제어하는 Daemon
- virt-manager = libvirt의 GUI 프로그램
- virt-viewer = VM(domain)의 모니터링 모듈
- virt-install = VM을 설치하는 모듈


#### KVM Host 서버에 Bridge 대역 구성하기
<img width="840" height="513" alt="image" src="https://github.com/user-attachments/assets/efe2761d-31f8-46af-89e8-645a8462ab76" />


1. 원래 이더넷 설정 바꾸기
   - vi /etc/sysconfig/network-scripts/ifcfg-<원래있던이더넷이름>
      ```
      TYPE=Ethernet
      DEVICE=eth0
      ONBOOT=yes
      BRIDGE=br0

      ※ 이 eth0은 br0의 브릿지 인터페이스로 사용하겠다고 선언.
      ```
2. 브릿지 이더넷 추가
   - yum install -y bridge-utils
     > bridge-utils 패키지가 없으면 만들어도 아무 동작하지 않거나 에러난다.

   - systemctl disable --now NetworkManager
     > 네트워크 매니저가 켜져있으면 브릿지 설정을 방해한다.
     
     
   - vi /etc/sysconfig/network-scripts/ifcfg-br0
     ```
     TYPE="Bridge"
     BOOTPROTO="none"
     DEFROUTE="yes"
     NAME="br0"
     DEVICE="br0"
     ONBOOT="yes"
     IPADDR="211.183.3.10"
     PREFIX="24"
     GATEWAY="211.183.3.2"
     DNS1="8.8.8.8"
     NM_CONTROLLED=no

     ※ 물리적 인터페이스가 있으면 br0라는 논리적 인터페이스로 브릿지 처리한다.
        원래 물리적 인터페이스가 하던 일을 br0가 대신한다. 하지만 실질적 트래픽은 물리적 인터페이스가 처리한다.
     ```
