## 1. git 놓친 부분 있는지 복습 진행

### git의 기본 구성
 1) Working Directory : 작업 공간
 2) Staging Area : 준비 영역
 3) Local Repository : 로컬 저장소
    -  "1. → 2. → 3." 순서로 진행

#### git 주요 CLI 명령어
1. git 연동 시작 : git init
2. git 구성 확인 : git config --list
3. git 상태 확인 : git status
4. git 커밋 대상 파일 스테이징 : git add 
5. git 변경 사항 확인 : git diff (커밋 전 검토 가능)
6. git 메시지 작성 & 커밋 : git commit -m
   - 커밋 메시지 구조) "태그: 설명" 형태이며, feat, fix, docs, test 등으로 태그 활용.
7. git 커밋 기록 한줄씩 : git log --oneline
8. git 브랜치 이동 : git switch "브랜치명" (브랜치 생성 & 이동 : git switch -c "브랜치명")
9. git 원격 저장소 연결 : git remote 
    - 확인 : git remote -v
    - 관례적으로, git remote add origin "주소" 형태로 최초 연결
10. 로컬 → 원격 연결 시, git push -u origin main (이후, 변경 사항은 git push만 해도 다 업로드됨)
11. 원결 → 로컬 변경사항 가져오기 : git pull origin main  (항상 pull 먼저 하는 습관 들이자.)

### HEAD에 대한 고찰
> 설계상 의도적으로 ① 브랜치를 가리킬 수도 있고 ② 커밋을 직접 가리킬 수도 있게 만들어졌다.
  - 개념 : "현재 작업 기준점"
     - 저장 위치 : .git/HEAD 이며, 해당 파일을 cat 명령어로 조회하면
  1) 일반적인 상황 : ref: refs/heads/main
  2) Detached HEAD(특정 커밋을 직접 가리킴) : abc12344s~~~~ ("커밋 해시"가 저장)
     - 발생 사유? : 특정 커밋 or 태그로 직접 체크아웃을 했을 때 발생한다.
     - 위험 요소? : 아무도 가리키지 않은 커밋은 가비지 컬렉션(gc)에 의해 자동으로 삭제될 수 있다.
       
### 커밋 vs 브랜치
1. 커밋 : 불변(immutable) / 모든 히스토리의 실체 / Git이 관리하는 실제 데이터
2. 브랜치 : 단순한 "포인터" 역할

### Pull Request (PR)
1. 정의 : main 브랜치 입장에서 로컬에서 내가 만든 브랜치를 "당겨서 가져가달라고 요청함"
2. 활용 예시
   - 로컬에서 브랜치 만들고, 수정을 하고 원격 저장소(ex. github)에 Push 하면, 특정 URL 링크 발생.
> 배움과 기록을 위해 리뷰는 구체적이고 친절하게 쓰자!

### gh 커맨드
1. github 레포지토리 생성 : gh repo create
2. github 레포지토리 복제 : gh repo fork (내 계정으로 복사본을 떠온다!)
3. github 레포지토리 자체 그대로 가져오기 : gh repo clone (주소창 복사해올 필요가 없어진다.)
4. github 이슈 올리기 : gh issue create -t "제목" -b "내용"
5. github PR 가져오기 : gh pr checkout "PR번호"
