## (1) ACM - AWS Certificate Manager
- TLS 인증서 발급 서비스. ***"HTTPS" 통신 시 필요!!***
- 별도로 Export를 하지 않는다면, ***특정 Region 에서만 사용가능***하다. (Export 기능이 과금이 굉장히 높다! 주의하자.) 

1. Domain names
   - FQDN을 입력, ***"asterisk(*)"*** 입력을 통해 ***모든 하위 도메인에 대해 다 인증서가 필요하도록*** 만들 수 있다!

2. 검증 방식 설정
   - DNS Verification : 간단한 CNAME 레코드 생성 후, 해당 레코드로 ACM이 접속 시도해서 검증하는 방식.

3. 검증 프로세스
   - Request가 완료되면, "Pending Verification" 상태로 들어간다.
   - Route 53을 통해 Record 생성해서 검증하는 방식이 가장 편하고 연동도 잘된다.
     > 만약 3rd-Party DNS 서비스를 사용한다면, CNAME을 복사해다가 등록하면 2~30분 내로 ACM이 자동 검증함!!


4. 활용 방법
   - ALB에서 Security Listener에 ACM을 선택해서 로드 밸런싱 방식으로 구현 가능하다.
   - 당연히 Security Group에서 HTTPS 통신 가능한 Inbound Rule을 추가해야한다. (443번 포트)
   - "Add Listener" 메뉴에서 "URL-Redirection" 설정 활성화를 통해 http 접속해도 https로 리다이렉션 되도록 설정하자.


## (2) CloudFront
<img width="858" height="462" alt="image" src="https://github.com/user-attachments/assets/08fa69f8-b992-44e5-b04f-e749ea2d0c4c" />
- CDN(Contents Delivery Network), 캐싱 서비스.
- Region과 Edge-Location 간 통신에는 AWS BackBone을 활용하며, 일반적인 트래픽에 비해서는 굉장히 빠르다.
- 첫번째 요청은 느리다. 두번째부터 Caching Server를 통해 미리 내가 사용하는 Region의 컨텐츠를 Edge-Location에 복사해둬서 멀리 떨어진 사용자에게 빠른 서비스 제공이 가능해진다!

 
1. TLS 인증서 발급
   - ***★반드시 "Virginia 북부(us-east-1) Region"에서 ACM으로 TLS를 만들자!!★***
     > Export를 하지 않았기 때문이다. 대신, us-east-1은 대표 Region이라서 글로벌하게 유효한 인증서로 사용할 수 있다!!


2. Specify Origin
   - S3, Elastic Load Balancer, API Gateway 등등 AWS 제공 서비스들을 붙여줄 수 있다.
   - Origin을 명시하는 건 마치 WEB 서버에서 Reverse Proxy 설정하는 것과 같은 개념이다.


3. Get TLS Certificate
   - 이때 Export한 TLS 인증서가 아니면 글로벌하게 유효하게 사용되는 us-east-1 Region에서 발행한 인증서를 써야한다.


4. Route domains to CloudFront
   - Route 53의 A, AAAA 레코드 등록을 자동으로 수행할 수 있다.
  

5. Origin - Behavior 변경
   - Default 값으로는 우리가 최초에 생성한 Origin이 등록되어 있다.
   - "Create Behavior"를 통해, Path Pattern을 커스터마이징해서 마치 REST api 연동 - ***Backend*** 단을 붙일 수 있다!!
     > 만약 Backend 로직을 연동한다면, Caching Policy는 Disable 처리하는 것이 좋다.
