## 1. AWS ECR(Elastic Container Registry)
- Host에 ***AWS CLI Plug-in 프로그램, Docker 패키지(또는 Desktop 프로그램)*** 설치가 필요함.
- AWS에서 제공하는 Docker Hub이며, ECR을 쓰려면 두 가지의 인증이 필요하다.
  ```
  - aws configure
  - docker login
  ```

1) ECR 내 Private Repository 생성
   - GUI 기반으로 생성을 해도 좋고, 아래 cli로 생성도 가능
     ```
     aws ecr describe-repositories
     # 레포지토리 조회
     
     aws ecr create-repository --repository-name <레포지토리이름>
     # 레포지토리 생성
     ```

2) ECR에 Repository로 이미지 Push
   - ***Registry 주소를 맞춰줘야한다!*** "docker tag" 명령어 쓰거나 image 만들 때부터 registry를 URI로 적용한다.
     ```
     docker push <AWS에서 할당해준 URI>/<레포지토리이름>:<태그>

     ※ 권한 없음 에러 뜰 경우
     aws ecr get-login-password --region <리전> \
     | docker login <Docker_ID> --username AWS --password-stdin <AWS에서 할당해준 URI>

     # AWS에서 토큰을 하나 발급받아서 표준입력(stdin)으로 비밀번호를 입력하겠다.
     ```


## 2. ECS(Elastic Container Service)
- AWS 제공 관리형 Container 서비스
- EC2의 AMI ~ Launch Template과 유사한 ***"Task"*** 를 사용한다.
  > Container Image + 포트정보, 리소스, 기타 추가 설정 등


- 당연히 Task들은 복제가 가능하며, 로드밸런싱도 구현 가능하다. ECS에서는 컨테이너를 "Service"라고 부른다.


1) Cluster
   - ***Container를 실행하기 위한 논리적 그룹***, 단순한 서버 묶음이 아닌 Resource + Scheduling 대상 + 관리 단위
     > EC2 기준으로는 Host들의 집합으로 생각하면 되고, Fargate 기준으로는 IaC 느낌으로 실행 공간 정도로 이해하자.


   - Fargate Only로 지정할 경우, Cloudformation에서 IaC 형태로 조회 가능하다. → ***"Stack"*** 이라 표현한다.


2) Task Definition
   - 위에서 이야기한 Launch Template 개념에 확장해서 중지된 Container 정도로 이해하자.
   - Application이 구동되기 위한 ***"포트", "Container image(ECR에서 불러올 수 있다.)를 여기서 지정!!***
   - 정의가 완료되면 Cloudwatch가 자동으로 활성화되어 모니터링이 가능하다.


3) Service
   - 위에서 정의한 Task Definition을 매핑해서 생성한다.
   - Deploy Configuration에서 기본값은 Replica이다. (Docker Swarm 할 때 복제본 만드는 설정이랑 똑같음)
   - 만약 EC2로 Cluster를 생성하면 Daemon 옵션이 선택 가능한데, 이는 Host 1개당 1개의 Task만 지정하는 옵션이다.
   - 당연히 생성할 때 Security Group 지정한다.
   - 보통의 경우 로드밸런싱이 필요하기 때문에 옵션에서 선택해줄 수 있다.
