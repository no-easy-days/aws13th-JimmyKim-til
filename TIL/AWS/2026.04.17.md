## 1. Lambda 개념 복습
 **(1) Function**
   - Runtime : 실행 환경. 제한적이긴 하지만 웬만한건 다 제공해줌
   - Handler Function : ***"가장 먼저 호출"*** 되는 함수. **event**(기본적으로 json 형태로 받아옴), **context** 2개의 인자값을 받음
   - Memory(최대 10GB), Timeout(최대 15분)까지만 설정 가능(클수록 과금 증가). CPU, 용량 등은 설정하지 않는다.
   
 **(2) ★Function Policy와 Execution Role 비교★**
   - Function Policy : ***"어떤 놈이 Lambda를 호출"*** 할 수 있는지에 대한 정책 ("Resource" 기반 정책, **"Principal"** 확인!!)   
   - Execution Role : ***"Lambda가 어떤 놈을 호출"*** 할 수 있는지에 대한 정책이 부여된 Role.

 **(3) ★Lambda 호출 방식★**
   - 동기식 (API Gateway, AWS CLI/SDK, Lambda 등등) 
      - Request하면 Response를 주는 형태이고, Status Code가 따라온다.
      -	오류 발생할 경우, Status Code 확인한 뒤 Application Code에서 재시도 로직을 설계 및 개발해야한다.
   - 비동기식 (S3, SNS, EventBridge 등등) 
      - Request하면 Event Queue에 쌓이고, Lambda가 순차적으로 처리한다.
	  - Response는 그냥 Event Queue에 할당됐다는거만 알려준다. **Lambda는 Event Queue만 관리**하고, 오류 발생 시 ***재시도 최대 2회 수행***.
	  - Lambda 작동 완료 이후 프로세스는 Destination에서 SQS, SNS, StepFunction, 다른 Lambda 함수 등을 지정할 수 있다.
   - 이벤트 소스 매핑
      - 지속적으로 ***Polling***(지켜보기)을 하고 있다가 이벤트가 발생하면 트리거 발동된 것 처럼 Lambda가 그때 실행됨.
	  -	Lambda를 직접 호출할 수 없다!! 그렇기 때문에 리소스 기반 정책을 주지 않는다.
	  - 데이터의 종류가 여러 가지 같이 들어올때 (e.g. DynamoDB) 활용한다.

## 3. Container 개념 복습
 - 소스코드 & 종속 라이브러리(버전 문제 多) & Runtime Engine & Config를 하나로 패키징. ***"Docker"***.
 - 이식성, 확장성이 매우 높다. ***★Stateless★***. MSA에 이상적인 모델.
 - 제일 먼저 "**컨테이너 이미지**(Dockerfile을 가지고 만든 Docker image)" 를 생성해야한다. **"Layer 단위"** 로 쌓아서 만들고, tag로 버전관리 필요.
 - AWS에서는 ECR(이미지 레지스트리), ECS & EKS(오케스트레이션 툴)를 제공하고 EC2 또는 Fargate를 통해 호스팅한다.


 **(1) EKS**  
   - K8s Control plane : ***API Server, Controller Manager, Cloud Controller, Scheduler, etcd 필요*** 
   - K8s Data plane : ***kube-proxy, kubelet, container runtime 필요***
   - 원래 Control Plane 모듈을 개별 설치해줘야 했던걸 AWS에서 전부 다 관리해주겠다는 개념. (Worker Node는 사용자가 관리)
   - AWS EKS는 전용 API(eksctl)를 활용해서 Control Plane을 관리한다. K8s의 API인 kubectl과 다르다.
   - 요즘 Pod 별로 설치해야하는 add-on이 너무~~ 많아져서 Auto-Mode가 도입되었다. (AWS의 개입 범위가 넓어짐)
     > K8s 자체 기술인지, AWS가 K8s를 관리하기 위해 개발한 기술인지를 구분 잘해서 공부하기!

   
 **(2) ECS**
   - EC2, Fargate로 구현하다. EC2 또는 Fargate로 구현한 ***인스턴스(≒Node)를 묶어 놓은건 "클러스터"*** 라고 칭한다.
   - K8s에서는 컨테이너를 묶어 놓은걸 "Pod", ECS에서는 컨테이너 묶어 놓은걸 ***"Task"*** 로 칭한다.
     - Pod, Task 별로 ***IP*** & 포트, 환경변수 등을 할당해준다. 권한이 또 중요한데... ***Task 별로 IAM Role을 할당***한다!
   - ***Task는 "단발성"*** 이다. Endpoint 역할 및 로드밸런싱을 위해 Pod 또는 Task를 묶어놓은 걸 K8s, ECS 둘다 ***"Service"*** 로 칭한다.
   - 이 Service들을 스케줄링해주는 걸 오케스트레이션이라고 부르며, ECS가 이 역할을 하는 것.

   
## 4. 배포 전략(Deployment Strategy)
 - 결국 CD(Continuous Deploy), ***★운영중인 서비스가 중단되면 안된다!★***
 - 배포 유형

   (1)In-Place
     - 서버 변경 없이 ***구동중인 서버에 그대로 배포***한다. 가장 비용 저렴하지만, 롤백이 매우 어렵다.
 
   (2)Rolling
     - 한방에 바꾸지 않고 구동중인 서버 중에 일부만 먼저 바꿔가며 ***순차적으로 배포***한다. 가동 중단 없음. 위험이 조금 완화됨. 
     - AWS Elastic Beanstalk 연동 가능하다.
   
   (3)Immutable
     - 잘 쓰지 않음. 기존 그룹을 없애버리고 새로운 그룹으로 옮긴다. 
   
   (4)***Blue/Green***(클라우드 환경에서 권장 사항)
     - Blue : 기존 운영 환경 그룹 / Green : 새로운 타겟 운영 환경 그룹. Green이 괜찮다는게 확인되면 Blue 그룹에 대한 트래픽 차단하는 방식.
     - 가동 중단 X, Green에 문제가 생기면 Blue가 살아있기 때문에 바로 롤백이 가능하다. 그러나 인스턴스가 남아있는거라 비용이 비싸다. 
     - ELB : VPC 내부에서만 가능. 그렇기 때문에 Blue/Green이 하나의 VPC 내에 있지 않을 경우 Route53 가중치 기반 점진적 배포.
     - CloudFormation(보통 Blue/Green에서 무조건 같이 씀), CodeDeploy & Elastic Beanstalk 연동 가능하다.
 
 
 - EC2 배포 옵션 : OneAtATime(하나씩 배포) / HalfAtATime(반씩 배포) / AllAtOnce(한방에 전부 배포) / 커스터마이징
 - ECS, Lambda 배포 옵션 : AllAtOnce(한방에 전부 배포) / Canary(한번 간보고 2번만에 배포) / Linear(점진적 배포)

## 5. CI/CD 테스트
 - 배포 전 스테이징 단계에서 테스트를 수행. AWS에 별도 CI/CD용 테스트 서비스가 없기 때문에 Open Source 쓴다.
 - 단위 테스트 → 통합 테스트 단계로 진행, 로드 테스트(과부하)도 하는게 좋다. 
 - A/B 테스트 : Route53의 가중치 기반 테스트, A가 AS-IS, B가 TO-BE 이고 UI 테스트 시 많이 활용함.
 - 인스턴스 Type 테스트 : Canary 형태로 단일 인스턴스에다 테스트 한번 해보고 그 이후에 실제 다중 인스턴스 만들고 배포하는 방식

## 6. 보안 자동화
 - DevSecOps. 보안을 한군데서 하는 것이 아니라 CI/CD 과정 및 자체를 포함하여 보안을 고려한다.
 - 결국 ***IAM으로 귀결***된다. 권한이 문제니까. Organizations로 계정에 대한 제어 정책 수립도 고려 대상.
 - IAM Access Analyzer : IAM User 별로 할당된 Policy가 혹여 문제가 있는지 없는지 분석해주는 서비스.
 - Global 레벨(Edge Location)에 대한 보안 서비스 :  WAF(L7), Shield(L3, L4)
 - AMI 및 컨테이너 이미지에 악성코드 같은 보안 이슈가 충분히 발생할 수 있다. "Inspector", ECR image scanning 서비스 활용하자.

## 7. 구성 관리
 - ***"AWS Config"***. 
 - ***리소스 변경 사항을 지속적으로 기록 & 추적***한다. 이걸 바탕으로 ***"규칙(Rule) 준수"*** 여부 확인. 어느 리소스가 규칙을 지키지 않았는지 알수 있음.
 - AWS에서 기본 제공하는 Managed Rule(관리형 규칙)도 있고 사용자 자체적으로 규칙을 만들 수 있다.
 - 어떤 리소스가 규칙을 지키지 않은게 확인되면, Lambda를 호출해서 Rule을 맞추게끔 교정 및 수정하는 간단한 Workflow를 만들 수 있다. 
    > CloudTrail은 "누가(IAM User)" 규칙을 지키지 않았는지 확인하다. 그러다보니 보통 Config랑 같이 세트로 묶어서 사용한다.

 - Log Event를 묶어서 Log Stream이 되고, Log Stream이 묶어서 Log Group이 된다.
