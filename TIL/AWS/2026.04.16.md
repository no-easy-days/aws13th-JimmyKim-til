## 📝 배운 내용 (Today I Learned)

## 1. 기초 개념 복습
 - 클라우드에서 가장 중요한 개념은 "Virtualization", "Utility Computing"이다.
 - 500번 에러에 대해서는 반드시 재시도(Retry)에 대한걸 고민하자.
 - HTTP Method에서 C(POST) R(GET) U(PUT) D(DELETE)를 잊지말고, 항상 ***Stateless*** 하다.
 - AWS는 기본적으로 Managed Service(관리형 서비스)이다. 
    - Instance 기반 : Type 선택(e.g. t3.micro) 필요, 가용성 고민 필수, 쓰던 안쓰던 Running 상태면 과금된다.
    - Serverless 기반 : AWS API를 호출하여 사용, API 호출하면 과금되는 구조
 - AWS "리소스"에 대해 권한(Policy)을 부여할 때는 IAM Role을 필수로 활용하는 걸 잊지 말자. 
    - Role 자체는 인"가" 관리만 할 수있기 때문에, STS(Security Token Service - 임시 AK 발급)를 활용해서 인"증" 정보를 관리한다.
 - Security Group은 "허용 규칙"만 추가할 수 있다. 기본은 아무것도 안들어가 있다 = ***All Deny***. (NACL은 All Allow, 차단 규칙만 추가할 수 있다)

 
## 2. Modern Application
 - ***"Application Integration"*** 이 필요하다. 방법은 API(동기식), Event-driven(비동기식 & 자동화), Message가 있다.
 - CI/CD : 애플리케이션 수정과 배포를 바로바로 하겠다. AWS에서는 CodePipeline, CodeBuild, CodeDeploy를 활용. 
 - IaC : 애플리케이션 배포 전에 인프라가 배포가 되어야 한다. AWS에서는 CloudFormation, SAM, CDK를 활용.
   - CloudFormation : yaml, json 활용하여 템플릿을 구성한다. Template → Stack → Resources
   - CDK : 각종 언어들(Python, Java 등)
   - SAM : yaml을 활용한다. 기본적으로 Serverless. SAM Template / SAM CLI 활용

   
## 3. 개발 도구를 사용한 CI/CD
 - 사람이 개입하면 지연이 발생할 수 밖에 없다. 결국 궁극적인 목적은 "애플리케이션 배포 자동화"
 - 소스 버전 제어가 필요한데, 보통 Git을 활용한다. (브랜치 따서 메인에 합치고 풀 리퀘스트 등등...)
 - AWS에서는 DevOps Agent를 선보였고, PR에 대해 코드 리뷰를 시킬 수 있다.
 - "Github Action" : Git Server에 소스코드를 커밋하자마자 CI Server(e.g. Jenkins)에서 서버 측 후크를 사용하여 빌드, 테스트, 프로비저닝까지 진행한다. 
    > 궁극적인 목적은 고객의 피드백 루프를 줄이기 위해서.


## 4. AWS CodeBuild
 - Jenkins와 같은 역할을 하는 CI Server. 소스코드 컴파일 & 테스트를 통해 SW 패키지(실행할 수 있는 형태)를 만들어주는 완전관리형 빌드 서비스.
 - Jenkins는 EC2 위에 올려서 써야하기때문에, Auto-Scaling을 고려해야한다. 대신, Jenkins 플러그인을 설치하면 AWS가 조금 도와줄 수 있음.
 - ***"★컨테이너 기반 환경★"***, 빌드 시작할 때 환경이 자동으로 만들어졌다가 빌드 끝나면 자동으로 없어진다. 설치하고 그짓거리 안해도되고 돈 아낄 수 있다. 
 - 빌드 아티팩트(결과물)은 보통 S3 Bucket에 업로드 되도록 설정하고, 당연히 CloudWatch Logs로 로깅할 수 있다.
 - ```aws codebuild create-project --generate-cli-skeleton``` : AWS CLI를 활용하기 위해, JSON 형태로 빌드 프로젝트 구조 생성하는 명령어	
 - ★buildspec★
   - yaml 파일, CodeBuild가 실행하는데 사용하는 빌드 명령이나 관련 설정의 모음이 들어있으며, 아래 4단계로 이루어진다.
	    > install(환경에 패키지 설치) → pre_build(빌드 전 명령) → build(실제 빌드 작업) → post_build(빌드 성공/실패 후 명령)

  
  - IAM Role을 활용해서 리소스에 엑세스 할 수 있는 Policy를 생성한다.


5. AWS CodeDeploy
 - Jenkins가 빌드 & 배포까지 해주는데, AWS에서는 CodeBuild랑 분리되어있다고 생각하면 된다.
 - EC2(***★AWS CodeDeploy 에이전트★*** EC2에 설치, ***Tags 활용한 필터링*** 필요!), Lambda, ECS(ELB, 클러스터 포함)에 배포 가능하다.
   > 실질적인 배포 과정은 에이전트가 수행한다!! K8s의 POD, ECS의 Task에 대한 엔드포인트 : "Service", Ingress 개념 복습하기!

   
 - ***★Event Hook★***과 함께 실행할 명령을 지정(appspec 파일 참조)한다. 배포 유형 옵션에서 In-Place(무중단)또는 Blue-Green(백업 느낌) 설정 가능.    
 - ★appspec★
    - yaml 파일, buildspec이랑 매우 흡사하게 생겼다. 아래 5단계로 이루어진다. ※ ApplicationStop 다음 단계인 DownloadBundle은 사용자가 제어 불가
	  - ApplicationStop(트래픽 중단 시 실행) → BeforeInstall → AfterInstall → ApplicationStart → ValidateService


## 6. AWS CodePipeline
 - 따로국밥처럼 쓰던 것들을 한데 모아서 자동화하는 느낌의 서비스. ***넓은 의미에서 IaC***로 볼 수 있다.
 - AWS Console로에서 작업해도 되고, json을 작성해서 AWS CLI로도 가능하다.
 - AWS CodePipeline의 6 Stage
    - ***소스(Git, S3, ECR등의 레포지토리)*** → ***빌드*** → 테스트 → ***배포*** → 호출 → 승인
    - 개발자가 git add ~ commit 이후 ***★git push 작업 시 트리거처럼 CodePipeline이 동작을 시작한다!★***
    - 소스 단계는 무조건 있어야하고, 빌드 or 배포 단계도 있어야한다. 그렇기 때문에 최소 2단계로 구성.
    - 각 단계에는 ***"Action"*** 이 할당되어있고, 한 단계에 1개 이상의 Action을 구성한다. Action은 병렬로 동작 가능하다.
    - 필요하다면, 수동 승인을 집어넣어서 검증 단계를 추가할 수 있다. 
       > 사람의 수동 검증 단계 전까지를 "Continuous Delivery"로 이야기한다. CI/CD에서 중간에 약간 껴있는 개념.
	
	
 - 여러가지 이유가 있겠지만 보통 ***"리소스 격리"*** 를 위해 Account를 여러 개로 나눠 만든다.
 - CodePipeline은 계정 간 공유 가능하다. A 계정에서 CodePipeline을 짜고 B 계정에서 배포 가능하다.
 

## 7. Micro-Service Architecture
 - ***★독립적(기술, 언어에 구애받지 않음)★***, 분산형, 한가지 기능에 특화, Black-Box, 구축~실행의 일원화.
 - 무조건 좋은건 아니다. 단점은 "복잡하다"는 것. 조직 환경에 맞춰서 선택해야한다. 그나마 이를 해결하기 위해 X-Ray 같은 추적 기능을 사용할 수 있다.
 - MSA의 정문 역할 : ***★API★ Gateway***.
   > API Gateway : Region Level! Global Level에서 구현되는 "CloudFront"랑 묶어서 쓰면 Edge Location 타고 AWS 네트워크 십분 활용하여 배포 가능!


## 8. Serverless Computing
 - AWS에서 인프라를 전부 관리 해주기 때문에, 확장성, 가용성, 보안 측면에서 유리하다. ***★VPC 바깥에 만들어진다!!!★***
 - 단점 : 자유롭지 못하다. 런타임도 정해져있고 사용 시간도 정해져있고 제약 조건이 꽤 많아 커스터마이징이 쉽지 않음.
 - Lambda : CloudWatch Logs 는 무조건 사용하게 강제된다.
 - Fargate : "컨테이너"용 서버리스 컴퓨팅 엔진.
 - AWS SAM
   - Cloudformation이랑 매우 유사한데, Serverless Application 구축을 위해 특화된 거라고 생각하자.
   - CloudFormation과 유사한 yaml을 템플릿으로 사용한다. "Transform: AWS::serverless-2016-10-31" 들어가있으면 SAM 템플릿. 
   - SAM에서 사용하는 ARN 구조
      ```
      AWS::Serverless::API
      # API Gateway

	    AWS::Serverless::Application
      # Application Repository 또는 S3 Bucket에 구현되어있는 Application.

	    AWS::Serverless::Function 
	    # Lambda
	 
      AWS::Serverless::LayerVersion
	    # Lambda에 필요한 라이브러리 또는 런타임 코드를 포함하는 Layer

 	    AWS::Serverless::SimpleTable
 	    # DynamoDB
	 
	    AWS::Serverless::StateMachine
	    # Step Function   
      ```
   - ```sam init``` : 기본적인 sam template을 cli 환경에서 가이드에 따라 초기화하여 만들 수 있는 명령어
   - ```sam build``` : 로컬 환경에서 ***"Docker 기반"***으로 소스코드 빌드(컴파일) 명령어
   - ```sam local invoke``` : Lambda를 Docker 컨테이너가 구동되는 로컬 환경에서 실행
