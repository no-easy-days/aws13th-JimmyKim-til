## Architecting On AWS

### 아키텍팅 기본 사항
1. 데이터 센터 : 수천대의 서버를 호스트함. 

2. AZ(가용 영역) : 하나 이상의 데이터 센터를 클러스터로 묶어둔 것.

3. 리전 : 3개 이상의 가용 영역 포함. 거버넌스, 지연 시간, 서비스 가용성, 비용 을 고려해야 한다.

4. 엣지 로케이션 : 글로벌하게 요청을 수신하고 더 빠른 전송을 위해 데이터를 캐싱함.
    > CloudFront : AWS의 CDN 서비스.

5. Local Zone : 리전과는 별도로, 은행의 지점 느낌으로 만들어서 사용자와 가까운 곳에 인프라를 구축해준다.
    > 사실 한국이랑은 크게... 관련 없고 미국, 러시아 같이 도시 간 거리가 매우 먼 경우에 활용

### AWS Architect의 역할
1. AWS Well-Architected Framework의 핵심 요소
  - 보안 / 성능 효율성 / 비용 최적화 / 운영 우수성 / 안정성 / 지속 가능성

2. Well-Architected(WA) Tool
   - AWS Well-Architected Framework에 기반하여, 어플리케이션 및 워크로드 검토하는 셀프 서비스 방식의 도구 (모범 사례 체크)

### IAM 복습
1. 사용자(User) : 인증을 위한 것 <--------------------------> 계정(Account)
  ※ AWS 접근 방법 3가지
    - Management Console : username & password
    - AWS CLI : Access Key(A.K) & Secret Access Key(S.A.K)
    - SDK(프로그래밍 언어 별) : A.K & S.A.K

2. 그룹(Group) : 1번의 사용자(User)를 용이하게 관리하기 위함. 말 그대로 그룹

3. 정책(Policy) : json 기반 문서로 권한 지정. 
   - 명시적 거부 : 요청에 대한 모든 정책 문서에서 Deny를 탐색하고 존재할 시 무조건 거부
   - 명시적 허용 : 요청에 대해 Allow일 때 같은 요청에 대한 명시적 거부가 없다면 허용한다.
   - 묵시적 거부 : 명시되지 않은 항목에 대한 모든 요청은 기본적으로 거부된다.
    1) 리소스 기반의 정책
        > "Principal" 확인! 대상을 정의하며, 있으면 리소스 기반이고 없으면 자격증명 기반이다.
    2) 자격증명 기반의 정책
         - Inline Policy (1:1) → 하나의 정책을 한명의 사용자에게만 지정. user 삭제 시 같이 지워진다.
         - Managed Policy (1:N) → 하나의 정책을 가지고 여러 사용자한테 부여하는 방식.
             - AWS Managed Policy : AWS에 의해 관리된다. 변경 불가
             - Customer Managed Policy : 사용자가 Tool을 이용하여 생성 & 활용. → Generator(GUI) 또는 Simulator(검증&테스트)
    3) 권한 경계(Permission Boundary) : 부여한 권한을 좀 더 축소하기 위한 기법, 관리형 정책을 사용하여 "최대 권한"을 설정한다.

4. 역할(Role) : 임시로 권한을 위임함. STS(Security Token Service), 임시 자격 증명(A.K, S.A.K 발행)과 같이 작동한다. ***모자, 또는 포스트잇***
    > 페더레이션 사용자 : IAM에서 직접 생성되지 않은 외부 사용자(Google 계정 등). AWS SSO 또는 AD 등을 활용해서 인증받는다.

### 다중 계정(Multi Account) 관리
1. AWS Organization : IAM보다 좀더 큰 개념. 계정마다 일일이 정책 부여할 필요 없이 **통합 관리**하는 기능.
  - 각 계정은 Organization Unit(OU)로 묶이며, SCP(Service Control Policy)를 통해 각 OU에 속한 모든 계정에 대한 권한을 제어한다.

### 네트워킹 복습
1. VPC (Virtual Private Cloud) : 가상의 격리된 네트워크 환경. 서브넷으로 나누어 사용한다.
  - 라우팅 테이블 : 네트워크 트래픽 제어. "길 찾기를 위한 지도"
  - 프라이빗 서브넷 : 외부 인터넷과 통신하지 않음. 로컬에서만 통신.
  - 퍼블릭 서브넷 : 퍼블릭 IP(외부 인터넷이랑 통신을 위한 IP) / 프라이빗 IP(내부에서만 통신을 위한 IP) 두가지를 가지게 됨!
  - 인터넷 게이트웨이(igw) : 외부 인터넷과 통신하기 위한 관문
  - 탄력적 IP(Elastic IP) : 외부 통신을 할 때 퍼블릭 IP를 할당 받지만 DHCP로 설정되어있기 때문에 매번 바뀌게 된다. 그걸 방지하기 위해, 어떤 인스턴스에서 다른 인스턴스로 IP 주소를 부여하는 방식.
  - NAT(Network Address Translation) : 기본적으로 VPC 내 프라이빗 서브넷은 폐쇄되어 있고, 외부 통신하고 싶으면 퍼블릭 IP가 필요하다. 그러나 항상 외부 통신이 필요한 건 아니기 때문에, 임시로 퍼블릭 IP를 할당해서 한시적으로 통신하게 해주는 것이 NAT. 여기서 탄력적 IP(Elastic IP)를 할당해서 사용한다.
  - 탄력적 네트워크 인터페이스(ENI) : 가상 네트워크 카드(NIC). 위에서부터 나오는 퍼블릭, 탄력적, 프라이빗 IP를 할당받는 곳.
  - NACL : "서브넷" 단위의 방화벽. 기본적으로 모두 Allow, ***Stateless***하며 Deny 및 Allow 설정 가능
  - Security Group : "인스턴스" 단위의 방화벽, 기본적으로 인바운드는 모두 Deny, ***Stateful***하며 Allow 설정 가능.
     ※ 보안 그룹 체이닝 : 소스 정보에 IP나 Hostname을 쓰지 않고, Security Group 이름을 바탕으로 설정하겠다는 의미.

### 컴퓨팅 복습
1. Elastic Compute Cloud(EC2)
  ※ 고려사항
   - 이름 및 태그 : 인스턴스를 식별하는 구분자. **대소문자를 구분**한다!
   - 애플리케이션 & OS 이미지(AMI) : AWS가 미리 구현해둔 템플릿도 있고, 직접 유저가 만들수도 있다.
   - 인스턴스 유형 명명 규칙 : "시리즈, ***세대***, 옵션, 인스턴스 크기" 순서로 명명되며, 여기서 ***세대*** 정보를 유의하자. 최신 세대는 성능, 확장성, 효율성이 좋다.
   - 키 페어 : 프라이빗 키(사용자가 저장) & 퍼블릭 키(EC2에 저장)로 구성된 보안 자격 증명 세트. 인스턴스 연결 시 이걸 활용해서 신원을 증명한다.
   - 테넌시 : 공유 테넌시(하드웨어 공유) → 전용 인스턴스(하드웨어 단위에서 격리된 구조) → 전용 호스트 (하드웨어를 독점함. 물리적 장비가 바뀌면 안되는 경우)
   - 배치 그룹 : 클러스터(인접하게 배치, 레이턴시 확보) / 분산형(세그먼트 형태, 내결함성 확보 목적) / 파티션(논리 그룹으로 분할, 상관관계를 고려하여 장애 방지)
   - 사용자 데이터 : 스크립트(유저가 설정한 OS 부팅&셋업 데이터 등)
   - 인스턴스 메타데이터 : 인스턴스를 구성 또는 관리할때 사용, 호스트 이름, 이벤트, 보안 그룹 등
  ※ Compute Optimizer : EC2 뿐만 아니라 EBS, RDS, Lambda 등을 포함하여 **비용 절감 & 성능 향상**을 위해 효율적인 리소스 구성을 제시함. 머신러닝 기반.

2. Elastic Block Store(EBS) --> USB 스틱을 생각하자. 
   - 가용 영역에서 자동 볼륨 복제를 통해 생성된 블록 스토리지. 만들어진 볼륨은 EC2에 연결해서 사용하며, 필요에 따라 다른 EC2에 붙일 수 있음.
   - 볼륨 유형 : SSD(gp2 < gp3 < io1 < io2 Block Express 순서로 고성능), 속도에 민감한 작업일 때 선택 / HDD(st1< sc1)
   - 인스턴스 스토어 볼륨 : 인스턴스에 물리적으로 직접 연결된 디스크 형태. I/O 성능은 빠르지만 인스턴스가 중단되면 접근이 불가능하고, 비영구적, 스냅샷 미지원.

3. Lambda --> "서버리스 컴퓨팅"
   - 인프라에 대해 관여하지 않아도 됨. AWS에서 관리해주겠습니다 느낌. ***함수***처럼 **이벤트 소스**의 "호출"에 의해 "서비스"가 구동된다. 
     > 이벤트 소스 : CloudWatch, S3, DynamoDB, ELB 등등이 포함되고, 다양한 서비스가 될 수 있음.

4. 인스턴스 요금 옵션
   - On-Demand : 기본이자 비쌈. 무제한
   - Savings-Plan : 기존 약정 요금제(Reserved)에서 조금 유연하게 완화한 요금제.
   - Spot Instance : 수요와 공급에 따라 가격이 변동한다. 내가 설정한 가격보다 시장 가격이 높아지면, 작동 중단될 수 있고 AWS가 몰수함.
