## 1. AWS REST API
 - API의 구조
   ```
   <GET & POST & DELETE & PUT & PATCH> http://<엔드포인트>/<리소스>

   ※ 리소스는 항상 명사형으로 쓴다.
     - httping(Linux) / tcping(Windows) : Latency 확인
   ```
 - ***모든 AWS Service는 AWS Service Endpoint를 활용해서 접속할 수 있다. 공식 문서를 구워 삶자.***
 - HTTP Status Code : 400번(Client Error), 500번(Server Error ~ AWS Service Endpoint, 이건 AWS 측 잘못)

 
 - AWS SDK : 언어별로 다 다르긴 하다. API를 더 편하게 개발할 수 있는 도구.
    - 자동 Retry 로직 보유, Pagination 지원. 
    - "Low-Level API" : 조금 더 REST API에 가깝게, 조금 더 "프로그램이 이해"할 수 있는 개념. → ***"Client!!"***
    - "High-Level API" : "사람들이 이해"할 수 있는 개념에 맞춰서 구현. → ***"Resource!!"***
    - aws configure 해줘야하는데, 보통 로컬 PC의 Home 디렉토리 아래에 .aws 디렉토리를 만들어 관리하는게 일반적임. --profile 등록하자.
    - OS 단에 환경 변수로 추가해서 쓰는 것도 가능하다. 

 
 - 기본적으로 2가지 작업을 한다.
    - 동기식/Blocking : 클라이언트가 작업을 요청하면 명령이 완료되기 까지 기다린다.
    - 비동기식/Non-Blocking : 클라이언트가 작업을 요청하면 기다리지 않고 다른 요청도 받는다.
                              그래서 지속적으로 상태를 확인하는 "웨이터"를 만들고(폴링), 요청을 처리 완료하기까지 기다리는 로직을 추가해줘야 한다.	

  
## 2. IAM을 통한 권한 부여
 - AWS CLI를 쓰기 위해 User & Key가 필요하고 이건 IAM에서 부여해준다.
 - 늘 배워왔듯이 User -> Group, "Policy & Role(임시적 권한. 주로 다른 계정 호출할 때)"을 부여하자.
   ```
   자격 증명 기반 정책에서 "Action"은 API를 호출하는 것을 의미하며 "AWS Service Endpoint"에 대한 접근을 의미한다.

   리소스 기반 정책에서 "Principal"은 "누가"(주체)이며, 자격 증명 기반 정책에는 없다.
   ```
   
 - 임시로 짧은 기간동안만 특별 권한을 부여하기 위해, 리소스에 "Role"을 연결할 수 있다. (EC2에 인스턴스 프로파일을 등록)
 
 
## 3. S3 스토리지
 - AWS 대표 제품. 객체 기반 스토리지. 디렉토리 구조가 아닌 prefix를 활용한다. S3 자체는 한번 지워지면 절대 복구 불가.
 - "정적" 웹사이트 호스팅(HTML, JS, CSS)이 가능한데, 이거 정말 잘 쓰이고 있다. "동적"은 불가능하고, "CORS" 기능 제공한다.
 - Bucket Versioning을 Enable하면, 똑같은 파일 이름이지만 버전을 다르게 만들어 백업 처리한다. (문제는 돈이 들어감...)
   - 삭제하게 되면 제일 마지막 버전만 삭제된다. Delete Marker에 의해서 지웠다는 표시만 난다. 버전 다 선택하고 마커도 선택해서 지워야 함.
 - 접근 제어는 "버킷 정책"으로 지정한다. 리소스 기반 정책이다.

 - 서비스 엔드포인트
   ```
   https://<s3 버킷 이름>.s3.<리전>.amazonaws.com/<prefix&리소스이름>

   ※ s3.<리전>.amazonaws.com 까지를 AWS Service Endpoint / <prefix&리소스이름> 을 "Key" 값으로 보면 된다.
   ```

 
 - API 구조
   > 당연히 동일하게 두 가지로 나뉜다. ① s3api : 하위수준(프로그램) / ② s3 : 상위수준 (사람)

  
 - AWS SDK 사용 방법 4가지 프로세스
   - .aws/config 파일을 생성할여 서비스 기본 정보를 구성한다.
   - python ~ boto3 패키지를 pip로 설치 필요하다.
	 - S3 클라이언트를 생성하는 로직을 개발한다.
	 - 서비스를 실행시킨다. (항상 마지막에는 세션 close를 하는 것이 좋다)


 - 객체 업로드 ~ "PUT"
   - Multipart Upload : 대용량의 데이터를 쪼개서(권장은 5GB 이내로), 오류날 가능성을 줄여서 업로드한다.
  
     
     ※ 프로그램이 조금 복잡해진다. 멀티쓰레드라던지 등등... 근데 S3 CLI 사용하면 기본이 MultiPart!(aws s3 cp)


 - 객체 읽기 ~ "GET"
   - getobject : 통으로 가져오거나 잘려진 객체를 불러온다.
  
     
     ※ HeadObject : 객체 "유무" 및 무결성 검증한다. (메타 데이터를 검색한다)


 - 객체에 대한 임시 접근 권한 부여
   - Public Access는 매우 위험하다. 그렇기때문에 미리 서명된 URL(presigned url) 방식을 활용하자. 배포 시간은 1분밖에 되지 않는다. 


## 4. DynamoDB
 - DBMS의 종류 ~ DB는 목적에 맞춰 적합한 DBMS를 선택하자.
   ```
   Key-Value DBMS : 고유 키를 사용하여 중복되지 않은 데이터를 직접 찾아간다. (RDBMS는 찾아가는게 오래걸릴 수 있다)
   Document DBMS : JSON 그대로 저장한다. JSON을 Query로 바꾸고 그런 짓 안해도 되는 이점이 있겠지. (MongoDB, Databricks)
   Vector DBMS : AI를 위한, Kubenetes 등등에서 로그 수집을 위해 많이 사용된다. (Elasticsearch, OpenSearch)
   ```
   
 - DynamoDB의 가장 큰 장점은 규모에 따른 성능이 언제나 일정하다. Latency를 보장한다.
 - 프로비전드(RCU, WCU) 방식 / 온디맨드(RRU, WRU) 방식 둘 중에 선택한다. 예측 가능한 비용을 고려할 땐 용량을 "미리" 설정하는 프로비전드 방식이 유리.
 - 주요 구조
   - 테이블에 Item을 등록한다. 파티션 키를 기본키로 사용하되, Sort 키를 같이 붙여서 고유 키이자 기본 인덱스가 되도록 한다.
   - 다른 속성을 가지고 데이터를 찾아가야 하는 경우에는 인덱싱이 안되어있기 때문에 Scan 필요하고, 이건 결국 RDBMS처럼 느려짐.
	 - 이를 방지하기 위해 보조 인덱스라는 걸 도입해서 쓴다. 
	    - Local ~  처음 테이블을 만들 때 지정할 수 있음, 그렇기 때문에 파티션키와 Sort키는 동일하다. 최대 5개
	    - Global ~ 이미 만들어진 상태에서도 가능함. 파티션키 & Sort 키가 달라질 수 있다.
 - API 구조
   - 세 가지로 나뉜다. 하위 수준, 상위 수준, 객체 지속성 수준
