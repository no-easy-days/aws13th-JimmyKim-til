## 1. EKS 주요 특징
- `Control Plane`(master node)과 `NodeGroup`(worker node)들로 구성됨.
  - `Control Plane`(master node) : aws에서 제공하며, 여기서부터 비용 발생한다.
  - `NodeGroup` : 관리형노드그룹(`Managed Node Group`) K8s 버전 자동 업그레이드. failover도 가능하지만 비싸다.

- 클러스터 생성 과정(약 10~15분 소요)


  <img width="807" height="700" alt="image" src="https://github.com/user-attachments/assets/05fc89fd-5eb5-4759-95a7-133657708765" />


  (1) 클러스터 생성
    - `eksctl create cluster...` 명령어 사용해서 `Control Plane`을 구성한다.
    - `eksctl`을 통해 생성하면 ***"cloudformation"*** 스크립트를 통해 클러스터 생성됨
    - `cloudformation` : AWS 제공 IaC, 클러스터 생성 중 에러 발생시 cloudformation에 가서 스크립트를 지워줘야함.
    - 만약 `terraform`으로 eks 클러스터를 구성한다면 cloudformation은 따로 생성되지 않는다.

  (2) NodeGroup 생성
    - `NodeGroup`을 만들어서 Worker Node들을 `NodeGroup`에 포함시킨다.
    - `NodeGroup`과 `Control Plane`이 통신이 되는지 확인후에 클러스터가 활성화된다.
    - Private Subnet에 Worker Node들이 존재한다면, Control Plane과 통신하기 위해 NAT-GW가 있어야 한다.

- 인스턴스 유형에 따라 Pod 최대 생성 갯수가 제한된다. (ex. t2.micro 는 Pod수 x개까지 만들 수 있다~)


## 2. EKS 초기 세팅
- On-Premise 환경준비 : 5가지가 설치 되어있어야 한다. `AWS CLI`, `eksctl`, `kubectl`, `kubeconfig`, `docker`

(1) AWS CLI
  ```
  apt-get -y install unzip
  curl "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip" -o "awscliv2.zip"
  unzip awscliv2.zip && ./aws/install

  aws configure       # Key입력, SecreyKey입력, Region입력, Format 세팅
  ```

(2) eksctl
  ```
  curl -fsSL "https://github.com/weaveworks/eksctl/releases/latest/download/eksctl_$(uname -s)_amd64.tar.gz" \
  | tar xz -C /usr/local/bin

  ※ 터미널 출력없이(에러는 보여주고) URL 접속 시 리다이렉션이면 그대로 따라가서
     그 결과로 stdout으로 오는걸 파이프 태워서 tar 명령어를 적용할텐데 현재 디렉토리에(-C .) 압축해제(xz)해라.
  ```

(3) kubectl
  - https://docs.aws.amazon.com/ko_kr/eks/latest/userguide/install-kubectl.html 공식 문서 참조해서 다운로드

(4) 명령어 자동완성 세팅
  ```
  # eksctl 작업
  echo 'complete -C "$(which aws_completer)" aws' >> ~/.bashrc
  eksctl completion bash | sudo tee /etc/bash_completion.d/eksctl > /dev/null
  eksctl completion bash >> ~/.bash_completion
  source ~/.bash_completion

  # kubectl 작업
  echo "source <(kubectl completion bash)" >> ~/.bashrc
  source ~/.bashrc
  ```

(5) eks cluster 구축
  ```
  eksctl create cluster --vpc-public-subnets <퍼블릭서브넷1>,<퍼블릭서브넷2> \
  --name <클러스터이름> --region <리전명> --version <쿠버네티스버전> \
  --nodegroup-name 노드그룹이름 --node-type <인스턴스유형> \
  --nodes <원하는노드갯수> --nodes-min <최소노드수> --nodes-max <최대노드수>

  ※ 주요 설정값만 할당한 버전, Public 대신 Private으로 하는 것이 일반적. (AWS VPC 사전 생성 필요!!)
  ```
  - `kubectl` 명령 에러 발생 시, `~/.kube/config`의 서버가 `api-server` 주소와 일치하는지 확인.
  - `aws eks update-kubeconfig --region ap-northeast-2 --name <클러스터이름>` : EKS와 맞춰주는 명령어


## 3. EKS 활용


(1) NLB
  - 별도 세팅없이 그저 `type: LoadBalancer`만 지정하면, AWS에서는 구닥다리(Classical LB)로 구현한다.
  - `yaml` 파일의 `metadata: ` 하위에 `annotations: `를 할당해서 Type을 지정한다.
    ``` 
    annotations:
    service.beta.kubernetes.io/aws-load-balancer-type: "nlb"
    ```

(2) **★ALB★**
  - 별도 Add-On 설치 필요. (Ingress-Controller 설치했던 것 처럼)


    <img width="1098" height="617" alt="image" src="https://github.com/user-attachments/assets/a1bfe574-dcfc-42ab-8eb6-d92a800ddd96" />


  - ingress 기술 적용 시 EKS에서 Load Balancer Controller(k8s의 리소스)가 ALB를 생성하는 구조이다.
  - 이를 위해선 Load Balancer Controller가 AWS에서 ALB를 생성할 수 있는 권한이 있어야한다.
  - 그럴려면 IAM Role 을 EKS의 SA에 부여해야하는데, 서로 다른 플랫폼이기 때문에 불가능하다.
  - 따라서 `OIDC`라는 매개체를 통해 `IRSA(Iam Role for SA)`를 생성하여 EKS의 SA에 assume(부여) 한다.
  - 그러면, 해당 SA는 AWS에 ALB를 생성할 수 있는 권한이 생긴다. 이 SA를 Load Balancer Controller에 부여한다.
  
  - 구현 과정
    ```
    1. JSON 형태의 AWS Policy 파일 다운로드
    curl -O https://raw.githubusercontent.com/kubernetes-sigs/aws-load-balancer-controller/v2.14.1/docs/install/iam_policy.json

    2. 다운로드한 JSON 파일을 통해 AWSLoadBalancerControllerIAMPolicy 생성
    aws iam create-policy --policy-name AWSLoadBalancerControllerIAMPolicy --policy-document file://iam_policy.json

    3. 유용하게 쓸 수 있는 환경변수 선언
    export CLUSTER_NAME=jimmy-eks-Cluster             #AWS Console - EKS에서 확인
    export ACCOUNT_ID=<어카운트번호>                   #aws sts get-caller-identity 명령으로 확인
    export REGION=ap-northeast-2
    export VPC_ID=<VPC ID> 				              #AWS Console - VPC에서 확인
    export ROLE_NAME==AmazonEKS_EFS_CSI_DriverRole    #Dynamic Provisioning을 위한 EFS-CSI Driver Role 지정

    4. OIDC 등록 및 활성화
    eksctl utils associate-iam-oidc-provider --cluster $CLUSTER_NAME --approve

    5. ALB 사용을 위한 IRSA(IamRoleServiceAccount) 생성.
    eksctl create iamserviceaccount --cluster=$CLUSTER_NAME --namespace=kube-system \
    --name=aws-load-balancer-controller --role-name AmazonEKSLoadBalancerControllerRole \
    --attach-policy-arn=arn:aws:iam::$ACCOUNT_ID:policy/AWSLoadBalancerControllerIAMPolicy \
    --override-existing-serviceaccounts --approve

    ※ kubectl get sa -n kube-system 명령어로 정상 생성되었는지 확인하자.
    
    6. Helm을 통한 aws-load-balancer-controller 설치 (당연히 사전에 Helm 설치되어 있어야함)
    helm repo add eks https://aws.github.io/eks-charts && helm repo update
    helm install aws-load-balancer-controller eks/aws-load-balancer-controller -n kube-system \
	  --set clusterName=$CLUSTER_NAME --set serviceAccount.create=false \
    --set serviceAccount.name=aws-load-balancer-controller \
	  --set image.repository=602401143452.dkr.ecr.ap-northeast-2.amazonaws.com/amazon/aws-load-balancer-controller \   	  --set region=ap-northeast-2 --set vpcId=$VPC_ID 
   
    ※ 이미지는 AWS 제공하는 리전 별 공식 레지스트리를 통해 다운로드한다.
       kubectl describe pod -n kube-system 명령어로 정상 생성되었는지 확인하자.
    ```


  - **"Ingress"** `yaml` 파일 예시
    ```
    apiVersion: networking.k8s.io/v1
    kind: Ingress
    metadata:
      name: "http-ingress"
      labels:
        app.kubernetes.io/name: "http-ingress"
      annotations:                                        #AWS Console에서 설정 지정하던 걸 여기서한다.
        alb.ingress.kubernetes.io/scheme: internet-facing
        alb.ingress.kubernetes.io/target-type: ip

    spec:
      ingressClassName: alb                         #ALB로 만들어라.
      rules:
        - http:
            paths:
              - pathType: Prefix
                path: /
                backend:
                  service:
                    name: svc-deployment            #당연히 service랑 맞춰줘야한다.
                    port:
                      number: 80
    ```

(3) Dynamic Provisioning
  - **"EFS(리전 공통)** 를 사용한다.
    > EBS는 특정 AZ에만 유효하고 특성 인스턴스에 종속되기 때문에, Static Provisioning 방식. 속도 빠르고 저렴하다.


  - 사전 세팅 및 설치
    ```
    1. 환경 변수 세팅 (OIDC는 활성화 되어 있어야 한다.)
    export ROLE_NAME=AmazonEKS_EFS_CSI_DriverRole

    2. EKS 다이나믹 프로비저닝 구현을 위한 SA 생성 (리전 공통)
    eksctl create iamserviceaccount --name efs-csi-controller-sa --namespace kube-system \
    --cluster $CLUSTER_NAME --role-name $ROLE_NAME --role-only \
    --attach-policy-arn arn:aws:iam::aws:policy/service-role/AmazonEFSCSIDriverPolicy --approve

    3. Add-On 설치. 여기서는 Helm 말고 eksctl 명령어 활용한다.
    eksctl create addon --cluster $CLUSTER_NAME --name  aws-efs-csi-driver --version latest \
    --service-account-role-arn arn:aws:iam::$ACCOUNT_ID:role/$ROLE_NAME --force
    ```
    
  - 이후에는 EFS를 AWS Console을 통해 생성해준다.
  - StorageClass `yaml` 파일 예시
    ```
    kind: StorageClass
    apiVersion: storage.k8s.io/v1
    metadata:
      name: efs-sc
    provisioner: efs.csi.aws.com
    parameters:
      provisioningMode: efs-ap
      fileSystemId: <AWS EFS ID>     # 여기서 매핑해줘야함!
      directoryPerms: "700"
    ```
  - PersistentVolumeClaim `yaml` 파일 예시
    ```
    apiVersion: v1
    kind: PersistentVolumeClaim
    metadata:
      name: efs-claim-1
    spec:
      accessModes:
        - ReadWriteMany
      storageClassName: efs-sc
      resources:
        requests:
          storage: 5Mi              # 용량 지정
    ```

(4) Secret 관리
  - 구현과정
    ```
    1. OIDC, 환경변수는 당연히 있어야한다.
    2. Secret CSI 드라이버 구현을 위한 SA 생성
    eksctl create iamserviceaccount --name secret-sa --region=$REGION --cluster $CLUSTER_NAME \
    --attach-policy-arn arn:aws:iam::aws:policy/SecretsManagerReadWrite \
    --approve --override-existing-serviceaccounts

    3.Helm을 통한 aws-secret csi driver 설치 (당연히 사전에 Helm 설치되어 있어야함)
    helm repo add secrets-store-csi-driver && helm repo update
    helm install -n kube-system csi-secrets-store --set syncSecret.enabled=true \
    --set enableSecretRotation=true --set rotationPollInterval=15s \
    secrets-store-csi-driver/secrets-store-csi-driver

    ※ aws에서 시크릿이 변경되더라도 동기화하는 옵션을 포함시켜뒀다.

    4. 토큰값 추가를 위한 csi-driver 편집
    kubectl edit csidriver secrets-store.csi.k8s.io 입력 후,

    tokenRequests:
      - audience: sts.amazonaws.com

    5. ASCP 설치
    kubectl apply -f https://raw.githubusercontent.com/aws/secrets-store-csi-driver-provider-aws/main/deployment/aws-provider-installer.yaml
    ```
  - SecretProviderClass `yaml` 파일 예시
    ```
    apiVersion: secrets-store.csi.x-k8s.io/v1
    kind: SecretProviderClass
    metadata:
      name: aws-secrets
    spec:
      provider: aws
      parameters:
        objects: |
            - objectName: "<시크릿의 ARN 주소>"
    ```
