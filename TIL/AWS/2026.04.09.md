## 1. DynamoDB 테이블 구축 과정
 (1) 테이블 설계
   - Access Pattern에 맞게 설계하자. Sort Key를 적절히 선택 & 올바르게 찾아가도록 설계해서 Scanning을 최소화하자. 
   - 인덱스를 적절히 설계하자. Secondary Index, 쿼리 분산 등으로 특정 부분에 I/O가 몰리지 않도록 설계하자.
   - ***"디스크 공간을 할당하지 않는다."*** 쓴만큼 돈을 낸다. 다만! ***데이터 I/O 용량(처리량)은 고려***해야한다.

 (2) 테이블 생성
   - DynamoDB 테이블 생성하는데, AWS CLI 를 활용해서 JSON 형태로 선언된 테이블 컨피그 파일을 활용하여 간편하게 생성할 수 있다.
   - JSON에 설계한대로 KeySchema ~ HASH(이게 Partition Key), RANGE(이게 Sort Key)를 적절히 지정하고, 속성값들을 지정하여 만든다.

 (3) 데이터 로드
   - "put-item" : 새 항목을 생성하거나 업데이트.
     > 하나의 item은 RRU의 최대 값인 400KB를 넘을 수 없다.

   
   - "batch-write-item" : 최대 25개의 PutItem & DeleteItem을 동시에 처리한다. 에러 처리는 필수적이다. 
     > Transaction API를 활용하면, 25개 중에 1개라도 문제가 생기면 롤백한다. (가격이 2배...)
   
   
   - "update-item" : 조건식으로 특정 아이템 단위로 업데이트한다. 아이템이 없으면 생성한다.    

 (4) 데이터 처리
   - "get-item" : 내가 원하는 걔만 가져와라! 기본 키(파티션 키, Sort 키)로 되어있는 "1개"의 아이템을 가져온다. 
   - "query" : 여러 개의 데이터를 가져온다. ***"기본 키(Partition key는 =, Sort key는 비교 연산자 활용)를 중심"*** 으로 조건을 가지고 검색한다.
     > 최대 1MB까지 검색할 수 있다! 당연히 결과값으로는 여러개의 데이터가 반환되고, 이거를 Pagination화 해서 볼 수 있다.

   
   - "scan" : 모든 데이터를 비교해서 찾는다. 느려지기도 하고 I/O가 굉장히 발생해서 비용도 매우 커짐. **BAD!!!**
   - "delete-item" : 항목 삭제. 위에 것들이랑 마찬가지로 조건식으로 아이템을 삭제한다.


 (5) DB 캐싱
   - DAX 사용한다. 당연히 App의 워크로드를 줄여줄 수 있다. VPC에 속한 App 기준 1ms 미만의 Latency를 제공한다.


## 2. AWS Lambda의 호출 모델
 (1) 동기식 : API Gateway를 통해 직접 호출한다. 재시도가 없다. 에러 생기면 끝나버린다.
 (2) 비동기식 : 잠깐 머물렀다가(대기열) 그 정보를 읽어와서 처리한다. 재시도를 2회 진행한다. S3, SNS을 활용한 푸쉬 방식과 연계한다.
 (3) Event source mapping : 마찬가지로 비동기식이다. Kinesis 소스배치, DynamoDB 소스스트림을 만들어서 연계한다.


## 3. AWS Lambda의 Runtime Environment
 (1) Cold Start
   - 소스코드를 실행시키면서 준비하고, 환경을 초기화한다. 프로그램 전체 품질에 영향을 주기 때문에 최소화하는 것이 포인트.
   - Colde Start 시간을 최소화 하기 위해, 특정 주기마다 이벤트 트리거를 걸어서 Latency를 최소화한다.
 (2) Warm Start
   - Cold Start가 마무리된 뒤 함수 핸들러를 실행한다. (실질적인 메인 로직이 구동되는 단계)
   - 지정된 개수 만큼 Warm Start 상태로 대기시킬 수 있다. (물론 돈이 들어간다...)
 (3) SnapStart
   - Java는 Cold Start 시간이 꽤 길다. 이걸 최소화하기 위해 ***"시작 환경을 Snapshot 처럼 저장"*** 한다. (물론 돈이 들어간다...)

 
## 4. AWS Lambda 권한 관리
 (1) 호출 권한(Role Base Policy) : "누가 Lambda를 실행할거냐"
   - 이벤트 소스가 ***"Lambda를 호출"할 때 필요***한 권한. 결국 리소스 정책.
 (2) 실행 권한(Execution role) : "Lambda가 뭘 실행할거냐"
   - Lambda 자체가 ***"AWS 서비스를 호출하거나 실행"할 때 필요***한 권한. 


## 5. AWS Lambda 개발 방법
 - AWS Console, AWS CLI, AWS SDK 또는 IDE에서 개발하고 ZIP 파일 묶어서(이건 언어 별로 차이가 있긴함, 최대 50MB) 업로드하는 방식 있음.
 - Lambda는 버저닝이 가능하며, Layer를 활용하여 배포 패키지 / 실제 로직 형태로 분리해서 구현할 수 있다. (아니면 공유할 때도 쓸 수 있음)
 - Stateless하며, Cloudwatch에 로그가 저장된다. (로그 쓸 수 있는 권한이 기본적으로 할당되어 있음)
 - AWS SAM을 활용하면 로컬에서 Lambda를 돌려보면서 테스트할 수 있다.
 (1) 핸들러 함수 ~ Args 로 event, context를 받음.
   - 이벤트 객체 : 호출할 당시에 받은 이벤트(=payload)
   - 컨텍스트 객체 : Lambda가 동작할 때 필요한 메타데이터 → "환경 정보(Runtime)"
 (2) 환경변수
   - "--environment" 활용.
 (3) 각종 API
   - "create-function" : Lambda 함수를 생성한다. 컨테이너 이미지일 수도 있다!(간단한 컨테이너는 굳이 K8s 안써도 되겠네. ECR 연계도 가능)
   - "update-function-configuration" : 기존 Lambda 함수 업데이트한다.
   - "invoke" : 동기식 또는 비동기식으로 Lambda를 호출한다.
 - 메모리는 최대 10GB까지, ***최대 런타임 시간은 "15분"*** 이며 timeout을 설정한다. 3000개까지 동시성(리전마다 다름) 지정 가능. 
 - 호출 페이로드(함수 호출 시 같이 보내는 데이터) : 동기식은 6MB, 비동기식은 1MB(SQS가 256KB...)이다.
   > /tmp : Lambda 내부적인 운영체제 디렉토리(Ephemeral Storage ~ 최대 10GB 까지 할당 가능)


6. AWS API Gateway
 - IAM User를 통해 Key값 받아 AWS Service Endpoint로 접근하는 전통적인 프로세스를 활용하지 않고 커스터마이징 할 수 있다.
 - ***결국 Request를 받아서 Validation → Mapping(Payload 적절히 변환)하여 Backend Service로 보내고, Backend로부터 받은 Response를 Mapping하여 리턴***한다. 
   > 프록시로 설정할 경우 Backend로 바로 던진다(Pass-Through). Mock으로 설정할 경우 Backend가 없다.

 
 - API Endpoint 설계, 생성, 게시, 모니터링 등을 할 수 있다. 종류는 HTTP, REST API, WebSocket, REST API private로 만들 수 있다.
 - 리소스를 설계하고, "Method 단위로 다르게 구성"할 수 있다. Lambda를 붙이거나 SQS, 커스터마이징 개발 API도 붙일 수 있고 "스로틀 제한(Throttling)"도 가능하다.
 - Swagger 또는 openAPI를 활용해서 API를 설계, 연동할 수 있다.
   ```
   ※ API 엔드포인트 구조

   "<api-id>.execute-api.<리전>.amazonaws.com/<Stage>"
   ```
 
 - 테스트를 원한다? AWS Console, AWS CLI, Postman 등을 활용할 수 있다. 
 - 배포를 원한다? Stage를 만들어서 개발/운영 구분해서 버저닝하고, Canary 릴리즈로 검증하자.
