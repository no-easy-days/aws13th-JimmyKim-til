## AWS Technical Essentials

### 항상 나오는 질문, 클라우드 컴퓨팅은 무엇인가?
***온디맨드, 인터넷, 종량제***

### Global Infra Structure (복습, 중요한 개념!)
1. 리전(Region) : 데이터 센터를 호스팅하는 지리적 위치. (3개 이상의 AZ를 포함한다)
   > 실제 서울에 있는가? → 그렇진 않음. 실제 위치는 아무도 모르지만, 누구나 아는 도시 이름으로 명명해서 인식하기 쉽도록.
2. 가용영역(Availability Zone) : 하나 이상의 데이터 센터를 "논리적 관점"에서 클러스터로 묶어둔 것. 
3. 엣지로케이션(Edge Location) : 전 세계 사용자에게 레이턴시를 보장하며 컨텐츠를 제공할 수 있도록 함. "캐싱"
   > 리전이 특정 지역에 국한되지만, 엣지로케이션은 아예 별도의 인프라! 용도 자체가 다르다.

### 서비스 접근, 관리 방법 (복습, 중요한 개념!)
1. AWS Management Console : 브라우저를 통해 GUI 환경에서 접근, 관리
2. AWS CLI : 터미널 & 커맨드를 통해 접근, 관리
3. AWS SDK : 특정 프로그래밍 언어의 라이브러리를 제공받아 접근, 관리

### 공동 책임 모델 (복습, 중요한 개념!)
1. 고객의 책임 : 클라우드 환경 "내부"의 보안
2. AWS의 책임 : 클라우드 환경 "자체"의 보안

### IAM(Identity and Access Management) - "인증", "권한" 관리
- Root Account : ***계정***, AWS의 모든 리소스 & 서비스에 완전한 액세스 권한을 가짐. 슈퍼 유저.
1. User(사용자) : 리소스 & 서비스와 상호작용하는 자격 증명
2. Group(그룹) : User의 묶음
3. Policy(정책) : 리소스 & 서비스에 대한 권한을 부여 or 거부 하는 문서. JSON.
    - Version(정책 언어의 버전) / Effect(액세스 허용 or 거부) / Action(작업 유형) / Resource(대상 객체)
4. Role(역할) : 리소스 & 서비스에 대한 권한을 "임시로" 부여

### AWS Computing
- EC2 인스턴스(일반적인 PC) / 컨테이너(어플리케이션이 돌리기 위해 모든걸 모아둔 것) / 서버리스
- AMI(Amazon Machine Image) : 인스턴스 초기화용 일종의 템플릿. OS, 스토리지 매핑, 권한 등을 가짐
1. EC2(Elastic Computing Cloud) 인스턴스
  ※ 온디멘드(Default - 무제한 요금제) / 예약(Reserved - 1or3년 약정 요금제) / 스팟(제일 저렴 - 변동 요금제) / Saving Plan(예약에서 조금 바꾼거)
  - 범용 / 컴퓨팅 최적화 / 메모리 최적화 / 가속 컴퓨팅 / 스토리지 최적화, 총 5가지의 타입 중에 골라서 구성함.
  - **보류중(준비단계), 실행중(시작됨), 재부팅중, 중지(Stop), 종료(Terminated)** 총 5가지의 상태값이 있음.
2. 컨테이너
  - VM : 각 가상 머신은 "가상화 플랫폼" 위에 개별로 "게스트 OS"를 가진다.
  - 컨테이너 : "컨테이너 플랫폼" 위에 존재하며, OS는 동일하다. 
  - AWS에서는 ECS, EKS등의 컨테이너 오케스트레이션 서비스를 제공한다.
3. 서버리스
  - 서버 프로비저닝 & 관리 필요 없음. 사용량에 따라 크기 조정되고, 사용한 만큼 비용 지불
  - AWS Lambda : **"함수"** 처럼 작동한다. 코드로 작성이 되어야하고, ***이벤트 소스에서 호출(Invoke) 해야 구동***된다.

### AWS Networking
1. VPC(Virtual Private Cloud)
  - 리전에 만들어진다. CIDR 표기법 기반 IPv4로 구성하고 기본적으로 VPC 안에서만 통신이 가능하도록 격리된 네트워크
  - 게이트웨이, NAT 구성을 통해 외부 인터넷 환경과 통신 가능
2. AWS Direct Connect
  - Direct Connect Position : 가산, 평촌에 위치하고 있는데, 여기까지 전용선을 끌어다 연결해주만 하면 나머지는 AWS가 전용선 연결 처리해줌.
3. 라우팅 테이블
  - 네트워크 트래픽의 전송 목적지 제어. 퍼블릭 IP 가 할당 된 상황에서 인터넷과 통신하기 위해 설정을 해준다.
4. 네트워크 액세스 제어 목록(NACL)
  - **서브넷 단위**의 방화벽. Stateless 하며 기본적으로 인바운드 아웃바운드 모두 Allow하며, **Deny 룰 추가**할 수 있다.
5. 보안 그룹(Security Group)
  - **인스턴스 단위**의 방화벽. Stateful 하며 기본적으로 인바운드 : 전부 Deny / 아웃바운드 : Allow하며, **Allow 룰 추가**할 수 있다. 

### AWS Storage
1. EBS(Elastic Block Store) "블록 스토리지"
  - SSD 형식 또는 HDD 형식으로 볼륨 유형 구성 가능
  - 스냅샷 : Incremental Backup(증분 백업, 마지막 백업 이후 변경 사항만 업데이트 하는 방식)을 지원함.
2. EFS(Elastic File Store) "파일 스토리지"
  - "NFS" 방식(리눅스라는 뜻)
3. FSx(File System x) "파일 스토리지"
  - 윈도우 환경에서도 쓰고 싶으니까. 다양한 운영 체제 환경에 맞춰 사용
4. S3(Simple Storage Service) "객체 스토리지"
  - 버킷의 이름은 고유 값으로 지정해야한다. 각 버킷 개별로 정책을 지정할 수 있으며, IAM 정책 지정 및 암호화 가능함
  - 스토리지 클래스 : Standard(Default) / Standard Infrequent Access(IA) / One-Zone Infrequent Access(IA) / Glacier Class
  - 클래스 선택 기준은 그 객체에 대한 ***액세스 빈도***를 보고 판단하며, Analyzer가 분석해준 사용 빈도를 참조하면 된다.
    > Standard가 제일 비싸고, 오른쪽으로 갈수록 요금 낮아지고, 파일 I/O 속도 느려진다.

### AWS Database : Architecting 과정에서 집중할 예정
- 관리형 데이터베이스 "RDS(Relational Database Service)" : 고객은 어플리케이션만 쓰면 되고 나머지는 AWS가 다 구성해줌. 고가용성.
- 비관리형 데이터베이스 : 7가지 제공 가능한 DBMS가 아닌 경우 직접 핸들링 해야하며, AWS는 인프라까지 제공.

### AWS 모니터링 및 로드밸런싱, 크기 조정
1. AWS CloudWatch
  - GUI형태로 시각화되어 유저에게 제공하는 성능 모니터링 서비스. ***"메트릭"(키-밸류쌍)***을 활용하며, 지표 / 로그 / 알람 구성 가능
2. ELB(Elastic Load Balancing)
  - 한 쪽으로 유저 요청이 몰려 부하를 주는걸 예방한다. 고가용성 & 탄력성 확보, 주기적인 Health-Check를 수행
  - "Listener" / "Target Group" / "Rules" 으로 구성되며, ALB(L7 Layer), NLB(L4 Layer), GLB(3rd-Party 솔루션 연동) 중 선택.
3. EC2 Auto-Scaling
  a) 수직적 용량 조정 방식 : Scale-Up / Scale-Down, 인스턴스의 스펙을 높이고 줄인다.
    - "***다운타임 발생***(서비스 중단 & 껐다 켜야함), ***최대 한도 존재.***"
  b) 수평적 용량 조정 방식 : Scale-out / Scale-In, 인스턴스를 추가하고 줄인다, AWS Auto-Scaling은 이 방식!
  - Launch 템플릿 : 인스턴스를 어떤 형식으로 구성할건지 문서화 & 버저닝하여 제공.
  - Auto-Scaling Group : Min, Max, 원하는 용량 값을 지정해준다.
