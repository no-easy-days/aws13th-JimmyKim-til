## <우리가 왜 이 클라우드 인프라 과정을 배우고 Cloud 기술을 공부하는가>
 - AWS & Data Center의 존재 의의
    - HW를 가상화해서 마치 SW처럼 만들어 ***"Service 형태"*** 로 제공. 가장 핵심은 지연 시간(Latency)이 짧아야 하고, 법적 기준을 준수해야한다.
 - Service 와 Resource의 차이점
    - ***Service는 "제품", Resource는 "실제 객체"*** 이다. (서비스:리소스 => EC2:Instance / EBS:Volume / S3:Bucket)
    - 만들어진 수천, 수만 가지의 Resource를 식별해야 하기 때문에 모든 Resource에는 ID가 붙여지는거고, 알아보기 쉽게 PreFix / ARN을 붙인다.
 - ★ AWS의 모든 서비스는 Level 별로 ***"저장 범위(Scope)"를 나눠 중복 저장하게끔 권장***한다.
    ```
    (1) Global Level : IAM, Route53, CloudFront
    (2) Region Level : VPC, S3
    (3) AZ Level : EC2, RDS
 
    ※ 인프라 Architecting 할 때 이 Scope를 생각하면 이해가 빨라진다. 보통 가장 중요하게 고려되는 사항은 VPC 내부인지 / 외부인지이다!!
    ```
 
 
 - IGW를 안쓰고 왜 VPC EndPoint를 왜 쓸까?
    - IGW는 학교의 정문이라면, VPC는 개구멍 뚫어놓는 것.
    - 외부 트래픽에 노출되지않고 안전하게 ***AWS 네트워크만 타고 통신할 수 있도록 하기 위해서***이다.


## <MSA가 왜 나온 것일까? 이 3일치 수업의 목적>
 - 비관리형 서비스 (Unmanaged Service)
    - 서버에게 필요한 특성은 ***(1)"프로비저닝"(준비 과정), (2)"가용성"(HA. 99.999999% 무중단 서비스 제공)*** 이다.
    - AWS에서 "1번만" 해주는 서비스. 대표적으로 EC2. 2번을 확보하기 위해 Auto-Scaling 등의 옵션 기능을 쓴다.
 - 관리형 서비스 (Managed Service)
    - AWS에서 "1번 & 2번 둘다" 해주는 걸 관리형 서비스라고 한다.
    - "Instance-Based" / "Serverless-Based"로 나뉘고, ***"Service Scope"에 따라 VPC 내부에 있는지/외부에 있는지 고려***하면 구분하기 쉽다!
      > 예시 : RDS 는 (CPU Spec, VPC, SG 등...) 지정 필요 → Instance 기반 / S3는 아무것도 선택하지 않음 → Serverless 기반(API 호출방식!).

	  
 - 마이크로 서비스 (MicroService)
    - 이전 방식을 Monolithic Service이라고 칭한다. Front-End, Back-end, DB 와 같이 3-tier.
    - Application 안에 여러가지 비즈니스 로직이 있는데, 하나의 비즈니스 로직만 수정&배포하더라도 그 App이 포함된 서버 자체를 Shut-Down해야 한다.
    - 당연히 배포 전 테스트 과정도 있을 것이고 혹시나 에러가 발생하면 리팩토링이 필요할 수도 있으며 전체 아키텍쳐에 영향을 주는 경우도 있다.
    - 이 불편함을 해소하기 위해, "비즈니스 로직(기능) 별로 분할"해서 운영하는 방식을 마이크로 서비스 아키텍처로 칭한다.
    - 대신, 비즈니스 로직끼리 통신은 해야하기 때문에 이 순간에 ***API / Event Driven을 활용한다. => "Application Integration"***								 
 
 
 - DevOps를 하는 이유
    - 특정 비즈니스 로직을 수정해야할 때 걔만 수정하고 배포하면 되는 ***MSA의 장점을 십분 활용한 일종의 "방법론"***.
    - ***개발과 운영을 동시에*** 한다. 그렇기 때문에 개발팀, 운영팀 간 장벽이 제거되어 ***비즈니스 로직 별로 조직이 구성***된다. (2-Pizza Team)
    - 인건비 절감 & 휴먼 에러 축소 및 운영 서비스 무중단을 위해 비즈니스 로직 코드 수정/배포 과정을 자동화를 시키는 걸 CI/CD로 칭한다.
    - 이제 비즈니스 로직을 뛰어 넘어 인프라 서비스에 대한 것까지도 코드로 만들어버려서 수정/배포 과정을 자동화 시키는걸 IaC로 칭한다.
	  > 결국 목적은 고객의 요청 사항 대응 & 피드백에 빠르게 반응하기 위해서.

## 2. 인프라 자동화(IaC)
 - 장점 : App 코드 처럼 버저닝 관리 기능 지원, 반복적이고 안정적으로 여러 환경 생성 가능
 - 구현 방법 : Shell Script / Application 코드 / CloudFormation / 3rd-Party 등으로 구현할 수 있다.
 - Stack : AWS에서 CloudFormation을 하나의 단위로 관리할 수 있는 리소스 모음.
 - ***CloudFormation "템플릿" 구조 (yaml 기반)***
   ```
   ---
   Description: 
   # 설명
   
   Parameters: 
   # Stack 만들 때 "입력" 받는 부분을 뜻한다. (t3.micro 쓸꺼야! 이런게 아니고, 값을 입력을 받음)     
      AllowedPattern:
      # 입력값 제한을 위한 표현식 지정
      AllowedValues :
      # EC2 인스턴스 유형과 같은 옵션 집합 지정. 예시≫ ['t2.micro', 't2.small']   
   
   Mappings:
   # Static 변수
   
   Conditions:
   # 리소스 생성 / 업데이트 시기에 조건을 부여
   
   Transform:
   # 매크로
   
   Resources:
   # 무조건 있어야 함! 제일 중요한 부분. 기본적으로 Resource는 병렬로 만들어진다. 사전 생성이 필요한게 자명한 것들은 AWS가 자동으로 관리한다.

     DependsOn:
     # AWS가 모르는 서비스에 대해 순서를 먼저 지정해야 하는 경우 추가로 옵션을 붙여줘야한다.
   
   Outputs:
   # 사용자 지정 리소스 값(URL, Name tag 등)
   ```

 
 - Stack을 생성할 때, SW 계층 단위로 나누어서 설계(계층화, "Layered"), 또는 Stack 안에 Stack을 불러오도록 구성(중첩, "Nested")하면 다른 곳에서 다시 써먹기도 좋다.
 - ***"드리프트 탐지(Detect Drift)"*** : CloudFormation 템플릿으로 "인프라 구성이 완료된 이후"에 AWS Console 또는 CLI에서 뭔가 수정을 했는지 확인하는 기능.
 - ***"변경 세트(Change Set)"***
    - aws cli ```create-change-set``` 명령으로 ***템플릿(yaml)을 수정***해서 일종의 ***"브랜치"***를 따는 기능.
    - 스택 업데이트를 하고 싶은데, 수정 사항이 기존 리소스에 대해서 얼마나 영향을 미치는지(예 : 중단여부 확인) 확인해준다. 							
 - ```CreationPolicy``` : Stack 생성 시 정상적으로 '준비 완료됐다는 신호'를 받을 때까지 특정 시간동안 기다리게 하는 옵션. (예 : Auto-Scaling 다 되었는지 등)


   ※ cfn-signal : ```CreationPolicy``` 결과값 바탕으로 CloudFormation에 성공/실패 플래그를 던져주는 헬퍼 스크립트. 모든 작업이 끝난 뒤 호출해야함!3

 - ```DeletionPolicy``` : Stack 삭제 시 리소스를 보존 / 백업 여부를 결정하는 옵션.
