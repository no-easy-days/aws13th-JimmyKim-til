- Security Group 생성 시, ***"Source"에 다른 Security Group을 적용***할 수 있는데 이를 Referencing 했다고 한다.

  
- 만약 Source에 특정 Security Group을 적용했다면, ***그 Security Group이 적용된 인스턴스로 부터 오는 트래픽에 대해 Rule을 설정***하는 것이다!

## (2) AWS Lambda
- Serverless. 서버가 없는건 아니고 AWS에서 만들어 준다 = "인프라 프로비저닝이 필요 없다" 
- Synchronous(동기식) 호출 방식, 결과가 반환될 때까지 기다린다.
  - Amazon API Gateway : 가장 일반적인 방식.
  - Function URL : Lambda 자체에 URL을 붙임. 간단하게 그 소스코드만 실행하면 될 때.
  - Self-Test : AWS Console에서 "Test" 탭으로 진입해서 직접 실행. 이벤트 발생시키는 JSON 작성해서 수행. (디버깅 용도)

### Lifecycle (Phase)
- AWS Lambda의 내부 동작 흐름.
- Init Phase, Invoke Phase, Shutdown Phase가 있음.

1. Init Phase = ***"COLD START"***
- 실행환경 구성, 코드 다운로드(S3, ECR), Extension(커스텀 모니터링, 보안...), Runtime Bootstrap(자바, 파이썬 등)
- Function Static Code 실행한다. 10초 제한이 있으며 10초 이후에는 invoke Phase에서 함수 실행 시간동안 init 수행.
- 처음 호출 시에만 이 Phase로 돌입하고, 이후에는 Invoke Phase에서만 실행된다.

2. Invoke Phase
- 실제 함수 실행 & 이벤트 처리 (handler를 실행)

3. Shutdown Phase
- 환경 정리하고, 5분 정도 실행환경을 유지한다. (캐싱)
- 코드 용량을 줄이거나, 라이브러리 Dependancy 최적화, SnapStart 활용 등을 통해 Cold Start 소요 시간을 줄이는 것도 신경써줘야 한다.


※ VM 환경에서 Lambda 파일 준비 프로세스


- Linux 환경에서도 Window에서처럼 똑같이 가상환경(venv) 구성해줄 수 있음! 당연히 Lambda는 venv로 세팅하는게 놓치는 부분이 없을 것.
```
apt install -y python3.12-venv
# venv 구축에 필요한 패키지 설치

sudo passwd root
# root 계정에 대한 패스워드를 설정

python3 -m venv /root/venv
# /root/venv에 venv module을 실행해라 = 가상환경을 구축해라.

source /root/venv/bin/activate
# venv 실행. 여기서부터 이제 필요한 pip를 설치하면 된다.

vi requirements.txt
# 필요한 pip들 명시한다.
★ Lambda는 Server를 띄우지 않는다! 그저 "handler"만 있으면 된다.
※ Mangum : Lambda Event를 ASGI(비동기) request로 변환함. 비동기 처리 필요 시 포함해야 한다.

pip install -r requrirements.txt -t <main, requirements 포함된 타겟 경로>
# -r : 패키지파일 목록 지정 / -t : 경로 지정

※ Lambda는 zip 파일만 받는다. 모듈 파일(main.py) & pip들 함께 담아서 넣어줘야한다.
```


※ 구현 완료된 Lambda를 AWS CLI를 통해 업로드
- CLI 실행할 OS에 AWS CLI 프로그램은 당연히 설치 되어있어야한다.
- Lambda Trust Policy를 적용해야하는데, json 형태로 미리 작성해둔다.
```
cat > trust-policy.json << EOF
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Service": "lambda.amazonaws.com"
      },
      "Action": "sts:AssumeRole"
    }
  ]
}
EOF
```
- IAM User 활용하여 AWS CLI 실행
```
aws iam create-role --role-name lambda-fastapi-execution-role --assume-role-policy-document file://trust-policy.json
# lambda-fastapi-execution-role 이라는 이름의 역할을 생성.

aws iam attach-role-policy --role-name lambda-fastapi-execution-role --policy-arn arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole
# Policy 추가 (로그 기록 볼 수 있는 권한)
  
ROLE_ARN=$(aws iam get-role --role-name lambda-fastapi-execution-role --query 'Role.Arn' --output text)
# 위에서 만든 Role의 arn을 ROLE_ARN 이라는 환경변수로 선언한다. (Lambda 생성 시 Role 지정 필요)

aws lambda create-function \
    --function-name zip-lam \
    --runtime python3.12 \
    --role $ROLE_ARN \
    --handler main.handler \
    --timeout 30 \
    --memory-size 512 \
    --zip-file fileb://lambda.zip \
    --region ap-northeast-2

# zip-lam 라는 이름의 람다함수 생성.
```
