### Custom AMI vs Launch Template
- Custom AMI : 인스턴스 유형이나 Security Group 설정이 포함되어있지 않다. OS 및 Package, Daemon만 설치되어 있다.
- Launch Template : Custom AMI + Security Group, Key-Pair, Subnet, IAM Role, EBS 설정 등과 같이 생성될 때 필요한 옵션들과 함께 묶여있다.
  > Custom AMI 정보에 추가적으로 '인스턴스를 생성할때 설정했던 내용'들을 미리 정의하기 위한 목적으로 생각하자.

- Auto-Scaling을 통해 Instance가 생성될 때마다 추가 설정을 일일이 지정해야하는데 현실적으로 불가능함... 그래서 Launch Template을 활용한다.


### Auto-Scaling Group 설정하기
- 주요 목적 : 고가용성(High Availability).
- Scale-Out : 서버 갯수를 늘린다. / Scale-In : 서버 갯수를 줄인다.
- Scale-Up : 서버 Spec(CPU, Memory 등등)을 높인다. / Scale-down : 서버 Spec을 낮춘다.
- 구동중인 서버를 건들지 않으려면 Scale In/Out 밖에 없다. 그렇기 때문에 Auto-Scaling Group에서는 이 방식을 채택한다.
- 구성 순서
  1. Elastic Load Balancer 생성
     - "Health Checks" → "Elastic Load Balancer Check(Recommended)"를 활성화!!
     - "Health Check Grace(유예) Period" → Health Check를 언제부터 시작할지 정하는 설정이다. 부팅 시간을 보상하기 위한 옵션.
  2. Target Group 생성 - Target을 비워둔 채로 생성!!
  3. Auto-Scaling Group 생성하는데, 이때 만들어둔 Target Group을 연동한다!!
     - "Auto-Scaling" → "Target Tracking Scaling Policy" 설정 활성화
     - "Monitoring" → "Enable group metrics collection within CloudWatch" 설정 활성화 (인스턴스의 지표를 CloudWatch에서 보겠다.)


- Auto-Scaling Group을 지우면, 그에 수반되는 ***Resource(EC2 포함!!!)들도 같이 삭제***된다.
