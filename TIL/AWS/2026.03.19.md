### 1. EC2(Elastic Compute Cloud)
- 가장 기본이 되는 서버. Key-Pair / Security Group 빼고 모든 옵션이 과금 요소!
- AMI 및 Instance Type은 되도록이면 최신 모델을 쓰자. (가격이 오히려 낮기도 하고, 운영 안정성도 높다)
- ID/PW 방식이 아닌 Key-Pair로 접속하자. (MobaXterm에서도 접속하기 위해 Private Key 발급 받아 로컬에서 접속! 키 관리 잘하자~)
- EC2에 붙어있는 저장소(Storage)는 앞서 배웠던 ***"LVM 기술로 구성된 EBS"*** 이다.
- 초기 설정 : Ping 조차도 불가능하다! ICMP도 허용되어있지 않다.
  > Security Group의 기본은 ★Deny All★ 이다. 처음 서버를 세팅할 때는 필요한 프로토콜을 고려해서 다 열어두자.


※ 계정 정보
  - Amazon Linux / CentOS / RHEL의 기본 계정(Root 권한 보유) : ***ec2-user***
  - Ubuntu의 기본 계정(Root 권한 보유) : ***ubuntu***
    > 항상 초기 접속 후, sudo -i 를 통해 root 권한을 얻고 서버 세팅을 시작하자.


### 2. VPC(Virtual Private Cloud)
- 독립적인 사설 네트워크. 각각의 VPC 끼리는 별도 설정이 없으면 당연히 통신이 안되어야 한다.
  - VPC Peering / VPN / Transit Gateway를 사용하면 각각의 VPC 끼리 통신 가능하게는 할 수 있다.
- 물리적으로는 떨어져 있지만, ***논리적으로는 같은 네트워크에 연결되어 있는 것처럼*** 네트워크 망을 구성해주는 기법.
- 가용 영역(AZ)는 웬만하면 A랑 C에서 작업하자. (모종의 이유로 B, D는 안되는 기능들이 몇개 있다고 한다.)
- VPC 생성 시, ***"Edit VPC Settings" → Enable DNS hostnames 활성화***하자!! 이것때문에 외부 통신이 안되는 경우가 있다.
- ***VPC 생성 이후, Filter를 잘 설정***해서 뒤죽박죽 섞이지 않도록 하자!!
- VPC 생성 이후 작업
  1) 첫번째 작업 : ***Internet Gateway(IGW)를 하나 생성하고, 연결***한다. (외부와 통신되도록 문을 만들자)
  2) 두번쨰 작업 : ***Subnet 생성***한다. Private, Public, DB 등이 3-tier Architecture 구현에 가장 기본 틀이다.
  3) 세번째 작업 : 각 서브넷 별로 ***Routing Table을 구분해서 생성***하고, ***Subnet Association****에서 연동시킨다.


※ Public Subnet 생성 시 주의 사항
 - "Auto-Assign IP Settings" 활성화 → 자동으로 Elastic IP(EIP)로 Public IP를 부여한다!
 - "Enable resource name DNS A record" 활성화 → WEB을 만들든 뭘하든 IP 접속 말고 FQDN으로도 접속 가능하도록 세팅한다. 
 - Routing Table(RT)에서 0.0.0.0/0으로 ***Default Route***를 잡아준다.


※ Security Group(SG)
 - 각 VPC 마다 개별로 설정 & 적용된다. 다른 VPC이면 그 VPC에 SG를 새로 만들어 줘야한다.


### 3. Bastion 서버에 대한 이해
- Private Subnet은 Routing Table 상에 외부(0.0.0.0/0)로 향하는 트래픽은 발생할 수 없다. 
- 이를 위해, Public Subnet에 존재하는 EC2를 거쳐서 Private Subnet으로 접속하게 만드는 역할을 하는걸 칭한다.
  <img width="1092" height="542" alt="image" src="https://github.com/user-attachments/assets/07856d19-a096-4f62-af81-c2345273de82" />


  - Bastion 서버를 만들었으면, SSH 접속을 통해 Private Subnet 안에 있는 인스턴스로 접근하자.
     ```
     ssh -i <개인키 저장 경로> <유저이름>@<Private IP주소>

     ※ Private Subnet은 Public Subnet과는 같은 VPC 안에 있기 때문에 통신이 가능하다! 물론 이것도 SG에 설정했기 때문에 가능.
     ```


### 4. NAT Gateway
- Private Subnet에 있지만 외부로 나가고 싶다. (Outbound는 가능하도록. Inbound는 당연히 X)
- 인스턴스가 Public에 있다면 EIP를 통해 받은 ***Public IP***도 있고, ***Default Route***가 적용되어 있으므로 외부로 나갈 수 있다.
- 이 점을 활용해서 ***Private Subnet → Public Subnet으로 PAT***하면 되고, 항상 ***NAT Gateway는 반드시 "Public Subnet"에 생성***해야 한다!!
- 그 후, Elastic IP를 Allocate 받아 IP를 하나 부여받도록 설정한다.
- NAT Gateway 생성이 완료되면, ***Private Subnet Routing Table에 Default Route를 "NAT Gateway"로 설정***하면 완료된다.
  
- ★주의★ NAT-GW는 과금 요소가 많다.
  > 시간당 기본요금 + 트래픽당 요금 + EIP가 복합적으로 과금되어 개비쌈.... 삭제 시에는 EIP 도 Release 해주자!!!


### 5. RDS
- AWS 제공 RDBMS. EC2보다는 비싸다. Aurora가 호환성은 좋지만 금액이 높기 때문에 MySQL 또는 PostgreSQL을 사용하자.
- 반드시 ***"Subnet Group"(두개 이상의 묶음)에 등록***해야한다. 반드시 두개 이상의 Az에 걸쳐서 만드는걸 AWS에서 강제하고 있다.
- 기본 접속 포트는 3306이겠지만, 보안 상의 이유로 바꾸는걸 권장한다.
- VPC에서 생성했던 RDS 용도의 Subnet 2개를 만들어 Subnet Group으로 할당한다.
- Security Group 설정 시, Inbound Rule 설정을 해야할 것이다. (Source를 Private에 적용된 SG로 하는게 가장 좋고, 당연히 포트 맞춰야함)
- 접속해보고 싶다면, Bastion 서버에서 SSH를 통해 Endpoint 주소를 입력해서 접속한다. (그냥 접속되면 DB 정보 다털리는 격)
  > mariadb-client 같은 DB 접속을 위한 클라이언트 패키지를 설치 필요.
