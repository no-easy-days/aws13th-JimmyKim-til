## 1. Cognito
 - IAM : AWS Service를 사용할 때 필요한 인증 & 권한 부여 서비스. 관리자가 IAM User를 Create 해줘야한다.
 - Serverless. Web 또는 모바일 앱에 대한 인증 및 권한 부여 & 사용자 관리 서비스. 

 (1) User Pool
   - 속성 : 개별 사용자의 메타 데이터(Email, 이름, 닉네임, 이미지)
	 - 그룹 : 사용자들 유형을 묶은 것
	 - 범위 : 리소스에 대한 Access Level 정의(읽기/쓰기 범위, OAuth 범위, 특정 사용자 등)
   - 위의 3가지가 구성이 되면, ***Cognito가 "Token - JWT" 발급하고 어플리케이션에서 이 Token 유효성을 검증***한다.
     ```
     ※ JWT Token : Base64로 인코딩되어 있고, Header ~ Payload ~ Signature로 구성됨.
        - ID 토큰 : 개별 사용자의 "속성" 정보가 들어간다.
		    - Access 토큰 : "인증 처리"해준다. 얘를 가지고 어플리케이션이 Cognito한테 인증을 요청함.
     ```

## 3. SAM ("Serverless" Application Model)
 - 개발 ~ 리뷰 ~ 배포 ~ 운영으로 이루어지는 전통 방식은 일관성도 없고 휴먼 에러도 발생할 가능성이 크다.
 - DevOps : 개발과 운영을 조화롭게 활용한다. 지속적인 전달과 지속적인 배포를 지향한다. K8s, CodeBuild, CloudFormation 등등의 도구가 있다.
 - CloudFormation에서 정의 & 생성한 "템플릿"을 활용하여, 특정 AWS 리소스(API Gateway, Lambda, DynamoDB, StateMachine)를 이용할 수 있게 한다. 
 - ***Serverless에 특화***되어있다. 로컬 환경에서 테스트 가능하며 아래 CLI 커맨드를 활용한다..
   - ```sam init``` : sam 초기화 (템플릿 가져오기)
   - ```sam build``` : 로컬 환경에 맞춰서 빌드
   - ```sam local invoke``` : Lambda를 Docker 컨테이너가 구동되는 로컬 환경에서 실행
   - ```sam local start-api``` : API Gateway Endpoint를 로컬로 복제
   - ```sam local generate-event``` : 이벤트 소스로부터 샘플 Payload를 생성

 
## 4. Observability(관측가능성)
 - 로깅 / 메트릭 / 추적 3가지를 바탕으로 구성된다.

   (1) CloudWatch (Logging & Metric) 
      - 각 언어 별 SDK / CloudWatch Agent를 EC2 또는 On-Prem에 설치 / AWS CLI 를 활용할 수 있다.
      - CloudWatch Application Insights : 어플리케이션을 검색하고, 문제를 감지하여 알림이나 액션을 설정할 수 있다.


   (2) AWS X-RAY (Tracing)
	  - "리소스 맵" 형태로 콘솔 상에서 한눈에 확인할 수 있다. 물론 세부적인 진행 정보도 추적 및 조회 가능하다.
	  - Lambda 및 API Gateway에서 Tracing을 활성화 해줘야 한다. 커스텀 APP이라면 SDK를 활용하면 될 것이다.
