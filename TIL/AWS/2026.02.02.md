## Architecting On AWS

## 2026.02.02 Well-Architected Best Practices
  > 클라우드 : Internet, Pay-as-you-go, On-Demand

1. AWS Well-Architected Framework

  - 평가, 감사의 목적이 아니다. 도움을 주는 도구이다. 1회성이 아니고, 지속적으로 적용한다.

  - ***운영 우수성, 안정성, 보안, 성능 효율성, 비용 최적화, 지속 가능성*** 5가지 요소로 구성.


2. 운영 우수성

   
  a. ***코드형 운영 수행*** ~ IaC(Infrastructure as a Code) → AWS CloudFormation등의 도구 활용
 
  - AWS System Manager : 일종의 "CM 도구", Configuration Manager. 설정하는거 다 들어 있음. 적용대상 : 서버(EC2 뿐만 아니라 물리적 실제 서버도 포함)
  
  
  b. DevOps에 대한 것을 잘 구현하자. 
 
  - **소스코드를 넘어, 배포의 범위에 Infra도 포함한다!**, AWS의 code pipeline, codeBuild, CodeCommit, CodeDeploy를 활용해보자. 
  
  
  c. 운영 절차를 빈번하게 재정의하자.

  - ID/PW 방식, Key Pair 방식(Key 잃어버리면 끝장, 털리면 다뚫림), AWS Session Manager(권장) 방식으로 프로세스와 절차를 점점 발전시켜 나가자.
  
  
  d. 장애를 예측하자.
  
  
  e. 모든 운영 실패로부터 미래를 대비하고, 학습하자.


3. 안정성


  a. 장애로부터 자동 복구되도록 설정하자. → ***"Auto-Scaling"***

  - AWS의 예시 : **EC2의 볼륨에 대한 Auto-Scaling** / RDS의 EBS 볼륨에 대한 Auto-Scaling / DynamoDB의 RCU, WCU에 대한 Auto-Scaling / Aurora의 Read&Replica에 대한 Auto-Scaling

  - 수직 크기 조정 : Scale-Up & Down, 기존에 있던걸 건드려야하기 떄문에 ***Down-Time 발생한다***.

  - 수평 크기 조정 : Scale-Out & In 권장. ***EC2 Auto-Scaling은 이 방식이다!!!***

  - 용량 추적이 불필요해서 편하다.

    
  ※ EC2 Auto-Scaling에 대한 고찰
  
   - 단점은 없을까? 단점을 생각해본 적 있나? → ***즉각적인 반응이 불가능하다. 만드는 시간때문에 항상 한박자 늦을 수 밖에 없다.*** 이 Gap을 줄이는 것이 포인트.
     
   - 약간의 기교가 필요하다. Event(Alarm, Trigger를 잘 커스터마이징 해야겠지) / Schedule(미리 부하가 걸릴 날짜 기반으로 지정) / Predictation(예측기반)에 대한 고민을 해보자.
   
   - GAP을 줄이기 위해.... 시작은 AMI(EC2의 type&spec, ebs, SG, keyPair, vpc(region, az, subnet), userdata 등을 담음), 다양하게 갖다쓸 수 있지만 전략에 따라 Gap이 달라지겠지.
   
   -  ***AMI의 Bake 전략*** 

  ① Row(AWS 제공하는거 쓰고, userdata 커스터마이징. userdata에 따라 EC2는 생성 시간이 영향을 받기 때문에 오래걸린다)

  ② Half(Row와 Full의 중간)
  
  ③ Full(AWS에서 제공하는 모든걸 다 쓴다. 속도가 빠르지만, AMI에 대한 관리 포인트가 증가한다. EC2 Image Builder를 쓰던 Packer를 쓰던 하면 됨)
  
  
  b. 복구 절차 테스트를 구현하자.
  
  - Fault Injection Simulator. QA 도구.
  
  
  c. 변경 관리를 잘하자.
 
  - AWS Change Manager 활용. 변경 관리 도구. 사실 이것보다 더 중요한건 ***Config(인벤토리 관리 서비스, Rule을 통해 관리)***.


4. 보안


  a. 강력한 자격 증명 기반을 구현하자.
  
  - IAM 제대로 쓰기 (계정, 사용자, 그룹, 정책, 역할에 대하여 명확하게!)
  
  
  b. 추적 기능을 구현해서 가시성을 확보하자.
  
  - SecurityHub(대쉬보드), GuardDuty(AI/ML기반 로그 분석해서 리포트 내줌), Inspector(서버 취약성 점검 도구)
  

  - CloudWatch(이벤트 감지 도구, ***Observability(Monitoring, Log, Tracing)***가 중요해졌다.), Config(Rule 지정), ***CloudTrail(API 호출 기록 도구)***, VPC Flow logs 등
  
  
  c. 각 계층에서 보안을 최대한으로 설정하자.
  
  
  d. 항상 사고는 사람으로부터. 지속적으로, 자동화하여 감사를 진행하자. 사람이 뭔가를 만지작거리는 걸 최소화하자.
  
  
  e. 전송과 저장에 대해 보안을 설정하자.
  
  - Macie(S3 대상 보안), KMS(키관리), CloudHSM, ACM(SSL 인증서 관리), SSE


  f. 특수 이벤트에 대해 대비하고 대응하자.
  
  - Detective(GuardDuty, CloudTrail 정보를 가져와서 분석 &리포팅), Config~규칙 적용(Lambda 트리거 거는 방식 등)


5. 성능 효율성


  a. 고급 기술을 대중화하자.
  - 혼자서 끙끙 앓지 마시고 AWS의 체~신 기술 한번 써보세요~
  
  
  b. 몇 분 안에 전 세계에 배포가 가능해야한다.  
  - 클라우드의 장점이자 쓰는 이유.
  
  
  c. 서버리스 아키텍처를 사용하자. 
  - EC2, 오토스케일, 로드밸런싱 등등으로 구현하면 관리 포인트가 너무 많으니... 서버리스로 구현하면 우리가 대신 관리해줄게~
  
  
  d. 새로나온거 써보고, AWS에서 제공하는 서비스를 쓰는 방법도 있다는 것을 잊지 말자.


6. 비용 최적화
- 클라우드는 쓴 만큼 비용을 지불(Pay-As-You-Go)하기 때문에, 불필요한 리소스를 최대한 찾아서 없애는 것이 가장 중요하다.
