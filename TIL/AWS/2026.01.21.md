## AWS Cloud Practitioner - 기본 개념
1. 용어 정리
  - 3 Tier Architecture : Presentation Tier(1계층), Logic(Application) Tier(2계층), Database Tier(3계층)으로 구성되고, **웹/앱/DB**로 이해
  - 프로비저닝 : 사용자의 요구사항에 맞춰 서버, 스토리지, 네트워크, 소프트웨어, 사용자 계정 등 IT 자원을 할당, 배치, 설정하여 사용할 수 있는 상태로 준비하고 배포하는 일련의 과정
  - 스케일 업(Scale-Up) : 성능 업그레이드 시 기존 하드웨어의 성능을 향상하는 방식. (온프레미스 : CPU, 메모리 부품 추가, 클라우드 : 더 좋은 인스턴스로 교체)
  - 스케일 아웃(Scale-Out) : 성능 업그레이드 시 그냥 서버를 추가하는 방식. (온프레미스 : 서버 교체 또는 추가, 클라우드 : 인스턴스를 더 늘리는 것)
  - 스케일 인(Scale-In): 스케일 아웃 했던 것의 반대. 자원을 다시 회수함. (온프레미스 : 서버 축소, 클라우드 : 인스턴스를 줄이는 것)
  - EC2 : "Elastic Compute Cloud" -> 클라우드 서버, 탄력적으로 오토 스케일링이 가능한 컴퓨팅 용량 활용.
    > Elastic : 탄력성, 늘였다가 줄였다가 자유자재로.
  - EC2 Instance : AWS에서 "서버" 자체를 칭하는 단어. 
    > Microsoft Azure : Virtual Machine, GCP : Compute Engine(또는 VM) Instance

2. 인스턴스 유형
  - 범용 : 컴퓨팅, 메모리, 네트워킹 리소스를 균형있게 제공
  - 컴퓨팅 최적화 :  고성능 프로세서
  - 메모리 최적화 : 고성능 데이터 베이스
  - 스토리지 최적화 : 데이터 웨어하우징 애플리케이션

3. 인스턴스 요금 옵션
  - 온디맨드(만들자마자 과금) / 스팟(짧은 시간만 사용) / 예약(1년 or 3년 약정) / 컴퓨트 세이빙 플랜(예약보다 좀더 유연하고, 온디맨드 비용에 비해 더 저렴) / 전용 인스턴스 (공유 자원 사용 X) / 전용 호스트 (아예 물리적 서버를 할당)

4. EC2 Auto Scailing
  - 컴퓨팅 요구 사항 변화에 따라 용량을 조절함. (동적 크기 조정 및 예측 크기 조정)
  - 수요가 적을 때 불필요한 EC2 인스턴스 제거 / 수요 많은 기간 / 수요 적은 기간에 EC2 인스턴스가 늘어나고 줄어듦

5. Elastic Load Balancing
  - 여러 리소스에 자동으로 트래픽을 분산, 수요 적을 때 vs 수요 많을 때 (Auto Scailing이랑 같이 사용하는게 일반적)
  - 여러 EC2 인스턴스에 워크로드를 분산 / 단일 EC2 인스턴스가 전체 워크로드를 처리하지 않아도 됨 / Auto Scaling 그룹으로 들어가는 트래픽의 단일 접점 제공

6. **가용 영역 (Availability Zone)**
  - 결국 "데이터센터"를 의미, 하나 이상의 데이터 센터가 포함되는 영역 (결국, 고가용성을 추구하기 위함)

7. **리전 (Region)**
  - 위의 가용 영역(AZ)을 "3개 이상" 묶은 경우를 의미.

8. **엣지 로케이션 (Edge Location)**
  - Amazon CloudFront(AWS 만의 "Contens Delivery Network" 서비스)가 가장 빠른 속도로 제공(캐싱) 하기 위한 일종의 "거점"
   > 네트워크 용어에서는 "Point of Presence(PoP)"와 유사한 개념.

9. AWS 서비스 활용 방법
  - Management Console(브라우저) / AWS CLI(터미널) / SDK(프로그래밍 언어 활용) 중 선택해서 활용하면 된다.
    > AWS 내부에서는 뭘 쓰든 다 똑같음.


## AWS 네트워크 개요
1. Amazon VPC (Virtual Private Cloud) : 고객이 정의하는 가상의 네트워크망. 퍼블릭 / 프라이빗으로 구성
  - 서브넷 : VPC 내부의 네트워크
  - 퍼블릭 서브넷 : 외부와 통신 가능 (보통 고객들이 외부에서도 접속이 가능한 "웹 서버"를 구성)
  - 프라이빗 서브넷 : 내부 끼리만 통신 (고객의 개인 정보가 포함된 "DB 서버", 비즈니스 로직이 있는 "Application 서버"를 격리할 때 구성)
  - 게이트웨이 : 외부 환경에서 VPC에 액세스하기 위한 접점
  - 보안 그룹(Security Group) : **EC2 인스턴스**의 "가상 방화벽", **"Stateful"** 특성을 가짐(인바운드 포트는 모두 거부, 아웃바운드는 허용)
  - 네트워크 액세스 제어 목록(Network Access Control List) : **서브넷** 내부의 "가상 방화벽", **"Stateless"** 특성을 가짐(인바운드, 아웃바운드 모두 거부)
  - 가상 프라이빗 게이트웨이 : 외부의 데이터센터와 VPN을 경유한 뒤, VPC 내부 네트워크에 액세스 하기 위한 접점
  - AWS Direct Connet : 외부의 온프레미스 데이터 센터 ~ VPC 간 전용 연결을 구현하기 위한 설정 방법. (VPN과는 다르다.)
  
2. AWS Route 53 : AWS에서 제공하는 DNS 웹 서비스.


## AWS 스토리지 개요
1. 블록 스토리지 : 파일을 동일한 크기의 데이터 블록으로 구분(예 : 물리적 HDD) **AWS : EBS 볼륨**(EC2 인스턴스를 만들 때 동시에 생성됨)
2. 객체 스토리지 : 파일 개별 개별로 구분 (예 : 구글 드라이브, 네이버 MyBox) **AWS : S3 스토리지**
   - 객체는 **"버킷"**에 저장되고, 객체에 대한 액세스를 제어하는 권한 설정할 수 있음. 
   - S3 Standard : 자주 액세스하는 데이터를 위한 클래스, 가용 영역 3곳에다 복사해서 저장함.
   - S3 Standard-IA : 자주 액세스하지 않는 데이터를 위한 클래스, 스토리지 가격은 낮지만 누군가 검색할 때는 가격이 높다.
   - S3 One Zone-IA : 가용 영역 "1곳에만" 데이터를 저장. Standard 클래스보다 가격이 저렴하다.
   - S3 **Glacier** *Instant* Retrival : 데이터 아카이빙 용도로 설계된 저비용 스토리지. Standard 보다 80% 정도 저렴.
   - S3 **Glacier** *Flexible* Retrival : 데이터 아카이빙 용도로 설계된 저비용 스토리지. 몇분~몇시간 정도 걸림. 
   - S3 **Glacier** *Deep* Archive : 데이터 아카이빙 용도로 설계된 저비용 스토리지. 12시간 정도 걸림, 제일 싸다.
3. 파일 스토리지 : 클라이언트가 공유 파일 폴더에 저장된 데이터에 액세스(예 : NAS) **AWS : EFS**


## AWS 데이터베이스 개요
1. Amazon Relational Database Service(RDS) : AWS에서 관계형 데이터베이스를 실행할 수 있는 서비스. (MySQL, PostgreSQL, MSSQL 등)
2. Amazon DynamoDB : AWS에서 비관계형 데이터베이스(도큐먼트, ***키-밸류 구조!!!***)를 실행할 수 있는 서비스. (예 : MongoDB 등)


## AWS 보안 개요
1. 공동 책임 모델 : 고객이 잘못했냐? AWS가 잘못했냐? 를 사고 유형에 따라 분류한 것.
    - 고객 : 인스턴스 운영 체제 / 호스트 기반 방화벽 / 애플리케이션 / 계정 관리 / 보안 그룹 등
    - AWS : 데이터 센터의 물리적 보안 / 네트워크 인프라 / VM 인프라 / HW or SW 인프라 등
2. AWS Identity and Access Management(IAM) : AWS 서비스와 리소스에 대한 액세스 관리 정책. 권한 부여하거나 막거나.
    - IAM USER : AWS 서비스 & 리소스와 상호 작용하는 사람 또는 애플리케이션을 나타내는 자격 증명, ***"최소 권한 원칙"***을 준수한다.
    - IAM GROUP : 유저 하나하나당 권한 부여하기가 귀찮고 복잡해서, 그룹 형태로 관리.
    - IAM ROLE : 권한에 액세스를 하기 위해 수임할 수 있는 **"임시 자격 증명"**
    - MFA(Multi-Factor Authentication) : "다중 인증", AWS 계정에 대한 추가 보호 계층. (2단계 인증이라고 생각하면 됨)
    > AWS ***Account : "ROOT USER!!!"***, AWS ***사용자 : IAM USER!!!***


## AWS 모니터링 및 분석 개요
1. AWS CloudWatch : 인프라 리소스 관련 모든 지표를 모니터링 하는 서비스. 대시보드 형태로 조회 가능.
2. AWS CloudTrail : 인프라 전체에 걸친 유저들의 API 호출 기록. "로그 저장" 서비스.
3. AWS Trusted Advisor : AWS 사용 환경을 검사하고 다른 AWS 모범 사례(레퍼런스)에 따라 실시간 권장 사항을 제시하는 서비스. 
    > Business Support Plan 채택한 사용자만 사용 가능


## AWS 요금 및 지원 개요
1. Free-Tier 범주 : 상시 무료 / 12개월 무료 / 평가판으로 구성
2. 요금 개념 : 종량제(사전에 용량을 프로비저닝하지 않고 사용한 리소스에 대해서만 비용 지불) / 예약 / 볼륨 기반 할인
   > 요금 계산기가 있음. 그거 쓰면 러프하게 알려주긴 함.
3. 통합 결제 : 여러 계정을 단일 청구서로 받아볼 수 있는 기능. (예 : 여러 S3를 하나로 묶어서 결제)
4. AWS Budgets : AWS 서비스 사용량 및 비용에 대한 임계값 설정 & 알림 서비스.
5. AWS Cost Explorer : AWS 리소스가 어디서 얼마나 사용되고 있는지 비율을 시각화하여 확인 & 관리할 수 있는 서비스.
6. AWS Support : Customer Support 엔지니어 직무가 수행하는 거. 총 5가지 플랜으로 구성
   - Basic Support(무료) / Developer Support / Business Support / Enterprise On-Ramp Support(상주인력... TAM이 붙음) / Enterprise Support
