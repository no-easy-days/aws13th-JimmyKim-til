## AWS Security Essentials - 기본 개념

### 클라우드 컴퓨팅의 3가지 주요 요소
***Internet / Pay as you Go / On Demand***

1. 보안 관점에서의 Well-Architected Framework
  - 자격 증명 및 액세스 관리 : AWS IAM
  - 인프라 보호 : VPC, WAF, Shield, Inspector
  - 데이터 보호 : KMS, Secrets Manager
  - 탐지 : CloudTrail CloudWatch, EventBridge
  - 인시던트 대응 : Incident Manager, Lambda

2. 공동 책임 모델
  - 고객 : 클라우드 **내부**의 보안을 책임진다.
  - AWS : 클라우드 **자체**의 보안을 책임진다.

3. AWS 인프라의 구성
  - 데이터 센터 → 가용 영역 → 리전
  > 리전 선택의 기준 : 규정 준수(특정 데이터는 특정 위치에만 저장), 고객 지연 시간(Latency, 고객과 가까운 곳), 비용(과금 정책이 다름)

4. AWS 컴플라이언스 센터
  - 각 나라 별 소프트웨어 운용 기준, 전 세계 인증 기관의 지침, Case Study, 모범 사례 등을 명시 & 제공
  -  AWS Artifact : 실제 문서(보고서, 인증서 등) 다운로드 기능 제공 서비스

5. AWS Config : 인벤토리 관리
  - ***"Rule"*** : 고객만의 규정을 만들 수 있고, AWS의 Rule을 잘 준수해야한다.


## AWS IAM 하이라이트
 a) User("사용자")
   - Management Console : ***ID***/PW 등록(루트 유저와 구분하자)
   - AWS CLI, SDK : Access Key / Secret Access Key
   ※ Account(***"계정"***, 루트 유저)와 구분해야 한다! ***Email***/PW로 로그인할 수 있고, 거기에 MFA(2단계 인증)를 추가해서 사용하자.

 b) Group
   - 사용자들의 묶음
    > "계정" 레벨에서 묶고 싶으면 AWS organization 서비스 활용하면 됨.

 c) Policy
   - 특정 사용자 또는 그룹에게 부여한 **권한들의 집합**, 
   - JSON 형태이며 managed(1:N) / inline(1:1) 방식이 있다. 정의 할 때 Effect, Action, Resource가 필수로 있어야 한다.
   - 사용자 단위 or 리소스 단위를 기준으로 권한 부여 여부를 결정한다. **(우선순위 : 리소스 단위 > 사용자 단위)**
     > "루트 계정" 레벨에 대해서 권한 부여하고 싶으면 AWS organization의 SCP를 활용하면 됨.

 d) ***Role("역할")***
   - 특정 사용자에게 **임시**로 어떠한 권한을 부여하는 방식. 모자를 썼다 벗었다하는 느낌.
   - 권한 정책과 신뢰 정책을 설정해야 한다. 결국 Policy와 매핑해서 씀!
   - IAM에 등록된 User에 대해서 할당이 되는거고, role이 할당되면 assume Role 된다고 표현함. 이건 결국 CloudTrail과 이어지는 거고, 보안 관제에 이점이 늘어남.
 

### 암호화를 통한 보호

 - 저장에서의 보안 : 암호화를 해야하고, "Key"가 필요. AWS 서비스 : KMS

## 네트워크 보안 개요

1. 리소스 격리의 순서→ 계정 > VPC > Subnet

2. VPC 아키텍처 개요
   - 계정 내 IAM 사용자가 VPC 구성 권한을 할당
   - 리전 선택 후 VPC 네트워크 구성. (AWS는 CIDR 기법 활용하고, /16~/28 까지 가능)
   - 인터넷 게이트웨이가 있어야 한다. (AWS가 직접 관리)
   - VPC 네트워크 내에 AZ가 존재하고, 그 안에서 서브네팅 시작. (대부분 Private / 용도에 맞게 Public 할당 가능)
     > 외부에서 접속이 되어야 하면(Public)? 웬만하면 ELB(ALB, NLB, GLB 중 선택)를 활용하는 것이 보안 상으로 좋다.
   - 라우팅 테이블, NAT 게이트웨이 구성. (NAT의 갯수에 따라 가용성이 달라진다.)

3. AWS 내의 보안(방화벽) 아키텍쳐
  - **Security Group** < **Network ACL** < **Routing Table**
  a) Security Group : Stateful, 전부 Deny 이고 일부만 Allow. ***"오직 Allow 정책만 넣을 수 있다"***
     - Security Group Chaining : 말 그대로 SG를 묶는 것. 
  b) NACL : Stateless, 일단 Allow. **Deny 룰을 추가할 수 있다.**

4. AWS WAF
  - 단독으로 쓸 수 없음. (CloudFront, API Gateway, ALB, Cognito, AppSync와 연동해서 써야 함)
  - 웹 ACL 생성 및 규칙 추가, 필터링 등의 방화벽 기능 제공
  - OSI 6, 7 Layer에 대한 공격 방어

5. AWS Shield
  - AWS의 DDoS 공격 방어를 위한 서비스. Standard : OSI 3, 4 Layer에 대한 공격 방어 / Advanced : WAF 기능 추가
  - 엣지 로케이션 레벨에서 동작한다.

6. AWS Route 53
  - AWS의 DNS 서비스, 꼭 써야하는 건 아니다.

7. AWS CloudFront
  - AWS의 CDN 서비스

8. AWS Direct Connect
  - AWS의 전용선 서비스

9. AWS CloudFormation
  - 보안을 표준화, 자동화 하는 Code를 템플릿화

10. AWS Firewall Manager
  - 계정에 대한 보안 관리, AWS Organization이 필수이며 WAF의 룰을 공유


## 서버 보안 개요
1. AMI : EC2 인스턴스 시작 시 필요한 모든 정보를 제공함
2. Amazon Inspector : EC2 대상 서버 취약점 진단 도구. 
3. AWS Systems Manager : 서버 관리 도구.

## 탐지 보안 개요
1. AWS CloudTrail : 로그 추적 도구. 
   > 얘만 있는건 아니다. 다른 서비스들(예 : VPC, WAF 등등) 안에도 로깅이 되긴 한다.
2. AWS Security Hub : 일종의 대쉬보드
3. AWS GuardDuty : ML 기반 각 서비스에서 취득한 로깅 정보를 바탕으로 분석 & 보호 기능 제공
4. Incident Manager : 일종의 장애 처리 메뉴얼. 
5. AWS Step Functions : lambda 뿐만 아니라 템플릿화된 코드를 모듈 느낌으로 GUI 형태로 짜는 것
