## 1. --link
- 대표적인 예시로 Application 과 DB 연동 시, DB가 IP를 부여받아 외부 노출 받는건 보안 상 별로 & 번거로움.
- ***Container의 "이름을 IP 주소처럼" 활용***한다.
  > --network 도 있긴한데, 일단 --link로 개념을 이해해보기


```
docker run --name <컨테이너 이름> ~~~(run 명령어) --link <링크할 컨테이너 이름> <레지스트리>/<레포지토리>:<태그>

※ 원리 : /etc/hosts 밑에 링크할 컨테이너 Host 정보를 등록해준 것이다. (미니 DNS 같이 이름으로 찾아감)
```


## 2. 개인 레지스트리 (private registry)
- 나만의 Docker Hub를 내 로컬 환경에 구축한다. 레지스트리 주소를 <대상IP주소>:<포트> 형식으로 등록.
- 레지스트리 주소에 맞게 컨테이너 이미지를 빌드해야 Push가 가능하다.

  
```
docker run --name <이름> --restart=always -dp <접속시킬포트>:5000 -v /registry:/var/lib/registry/docker/registry/v2 registry:latest

※ 컨테이너 중지됐을 때 자동 재시작 설정을 걸어줬다.
  - /var/lib/registry/docker/registry/v2 : 컨테이너 이미지가 push 되었을 때 저장 경로
```


- 기본적으로 docker push는 https 통신을 하게 되어있다. 아래 두가지 방법 중 하나로 구현해야 한다.
  - TLS 인증서를 붙이기 (당연히 인증서 필요)
  - http 통신으로 변경 설정 적용
    ```
    vi /etc/docker/daemon.json
    
    {
        "insecure-registries" : ["<대상IP주소>:<포트>"]
    }

    ※ 클라이언트(Push)로서의 설정. Docker는 시작 시 해당 파일을 읽어들인다!
    ```


- habor 나 nexus3 같은 것들이 이러한 기술을 활용한 것이다.


## 3. 대표적인 App 배포해보기

(1) Framework : "Application은 이렇게 만들면 됩니다~" 같은 일종의 가이드 & 기능을 제공하는 Tool.
   - JAVA 계열 대표 Web App. Framework : spring, springboot (JDK 설치 필요)
   - PYTHON 계열 대표 Web App. Framework : flask, Django, fastAPI (Python & 일부 pip 설치 필요)
   - Javascript 계열 대표 Web App. Framework : node.js (Framework라고 하기엔 약간 애매... npm 설치 필요)


(2) Build & Packaging의 개념 : 각 Framework로 소스코드 개발을 하면 App 배포를 위해서는 Build 필요.
   - springboot로 예를 들면, ***"maven, gradle"*** 같은 툴을 활용해서 jar / war 파일로 빌드한다.

     ※ 물론 Java 버전은 호환성 맞춰줘야 함.


   - nods.js로 예를 들면, ***"nodejs, npm"*** 을 활용해서 빌드한다.
     > nodejs : 개발 환경, npm : python의 pip와 비슷한 개념


(3) Dockerfile 작성 팁
   - 앞서 필요한 빌드 & 패키징 작업에 대해 Dockerfile 작성 시 ***RUN / CMD를 적절히 활용해서 배포한다!!!***
   - 원래 환경에서 명령어 입력해서 진행되던 걸 컨테이너 단으로 옮겨온 것 뿐이다. 어려워하지 말자.
   - ***".dockerigore", ".env"*** 파일을 활용하면 환경변수나 Container 생성 시 불필요한 파일들을 포함하지 않는다.
     ```
     ※ 대표 예시 ~ java springboot
     1) RUN mvn clean package
        - maven을 활용해서 소스코드를 jar 파일로 App 빌드 & 패키징

     2) CMD ["java", "-jar", "<대상파일경로 ~ App.jar>"
        - jar 파일을 서버에 설치된 java runtime을 활용하여 App 실행 

     ※ 대표 예시 ~ node.js
     1) RUN npm install
        - 의존성 패키지 목록(packages.json)이 필요, 실행 시 node_modules 경로에 의존성 패키지 설치

     2) CMD ["node", "<대상파일경로 ~ app.js>"]
        - js 파일을 서버에 설치된 node.js 환경을 활용하여 App 실행

     ※ 대표 예시 ~ Python flask
     1) RUN pip install -r requirements.txt
        - 의존성 패키지 목록(requirements.txt)이 필요, 참조하여 site-packages로 pip 설치

     2) CMD ["gunicorn", "-b", "0.0.0.0:8080", "<실행파일 ~ main:py>"]
        - python 파일을 서버에 설치된 gunicorn 패키지를 활용하여 App 실행
     ``` 

## 4. Multi-Stage Build
- Dockerfile을 여러개의 Stage 형태로 일종의 Layer를 분할하여 작성한 기법.
- java springboot로 예를 들면...
  <img width="1080" height="673" alt="image" src="https://github.com/user-attachments/assets/5bbce1ec-168d-40c8-99bb-586b51734fde" />


- FROM을 두번 해준다.
  ```
  # 첫번째 stage
  FROM maven:3.9-amazoncorretto-8 AS builder
  # "builder"라는 이름으로 명명 & Build-Stage로 선언. 이름은 내 맘대로 하면 된다. 

  WORKDIR /app
  COPY . .
  # 호스트의 Dockerfile 저장 위치에 있는 모든 파일이 컨테이너 내부의 /app 에 복사

  RUN mvn clean package
  # target 폴더에 springbootApp.jar 생성

  # 여기부터 두번째 stage
  FROM public.ecr.aws/docker/library/openjdk:8-jre-slim
  WORKDIR /app2
  COPY --from=builder /app/target/springbootApp.jar app.jar
  # 이전 "builder"라는 스테이지에서 생성된 파일을 현재 이미지로 복사. 위의 이름을 맞춰줘야함!

  CMD ["java","-jar","app.jar"]  
  ```

    
  > Stage 갯수는 제한이 없다. 상황에 따라 Multi를 쓰거나 단일로 하거나는 엔지니어가 판단하는 것!
