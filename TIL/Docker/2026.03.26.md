## Container Virtualization(가상화)
<img width="1085" height="632" alt="image" src="https://github.com/user-attachments/assets/dd2d206d-0667-4624-be49-ab39b9301b77" />


- ***Host OS의 Kernel 단***까지 공유해서 쓰고, 나머지 전부 가상화한다. 각 컨테이너는 Kernel Level에서 분리된다.
- 그렇기 때문에 Guest VM & OS 구성 작업은 필요 없고, APP(종속 패키지들도 함께 패키징해야함) & Shell만 잘 구성하면 됨.
- ***프로세스 수준의 격리, Resource Overhead 낮음, 보안성 상대적으로 취약, 프로비저닝 속도 빠름***
  - Hypervisor 가상화 비교 : ***하드웨어 수준의 격리, Resource Overhead 높음, 보안성 양호, 프로비저닝 속도 느림.***
- 이식성(Portability)이 Hypervisor에 비해 매우 높다.
- 원하는 Kernel을 선택할 수 없기 떄문에, ***★Host Kernel에 호환되는 환경만 구성 가능★***하다. 
  > Linux : Linux 기반 Container, Windows : Windows 기반 Container(단, Docker Desktop 사용 시 Linux 기반 Container 구현 가능함!)


- Docker : 가장 널리 쓰이는 컨테이너 가상화 기법 / Containerd : K8s를 구성하는 Runtime. (그 외 : CRIO, LXC)


### Linux kernel based Container 가상화 기반 기술 3가지
1. chroot : 최상위 디렉토리를 변경.
   - 컨테이너의 최상위 디렉토리를 바꿔서 ***컨테이너 내부에서 접근 가능한 디렉토리를 제한***한다.
2. cgroup : Resource(CPU, Memory, Disk 등등) 컨트롤.
   - 컨테이너에 호스트의 Resource를 ***"할당"이 아닌 "제한"*** 한다. 리소스를 떼어다 주는게 아님!
     > 여기까지만 쓸수 있어~~ 같은 느낌. Host의 리소스 다 땡겨 쓰는걸 방지하기 위해.


3. namespace : PID 관리.
   - Process, 네트워크, File 등을 Host로 부터 격리한다. 당연히 Host에는 접근하게 하면 안되겠지?
     > 컨테이너가 별도로 관리하겠다!!

1. watch docker ps
   - "실시간"으로 동작중인 컨테이너 조회, -a 붙이면 종료된 것까지 전부 조회한다.
  

2. ### docker run
   - Container image를 기반으로 생성(create) & 실행(start) 명령을 동시에 수행한다.
   - Container image는 여러 개의 Layer로 분할된 형태로 구성되어 있으며, image 자체는 항상 일정하게 유지된다.

   ```
   docker run --name <원하는이름> <레지스트리>/<레포지토리>:<태그> <컨테이너 run 이후에 수행할 Command>
   # 이렇게만 입력하면 그 "컨테이너 내부로 진입"한다. ctrl + c 로 나가면 그 컨테이너는 종료된다.
   # 태그를 명시하지 않으면 default는 latest이고, 레지스트리의 default는 docker.io이다.
   # run 이후 수행할 Command는 무조건 foreground로 동작해야한다!!!

   ※ 주요 옵션 파라미터
   -d : "Detach", Host 입장에서 마치 Daemon처럼 Background로 컨테이너를 실행시킨다. ★★★★★
   
   -v : "Volume", Container의 특정 디렉토리를 Host의 특정 디렉토리에 마운트한다. ★★★★★
        컨테이너가 Stop되어도 파일은 Host에 남는다.
        우선순위는 Host 마운트 포인트가 더 높아서, Container 마운트 포인트에 파일이 있어도 무시된다.
        무시된 파일은 없어지지 않는다!! "Overlay"되어서 가려진 것 뿐이다.
        이 옵션은 여러번 쓸 수 있다. 마운트 포인트를 여러개 만들어둘 수 있다는 이야기.

   -p : "Publish", Container를 외부에 노출시킨다. 마치 DNAT처럼. ★★★★★
        IP 특정 필요 없다! → docker run 명령은 결국 Host에서 내려주는거고, 명령하는 순간 대상이 특정되기 때문이다.

   -e : "Environment", 환경변수를 구성한다. 특정 Container는 "필수 환경변수"가 있는데, 이 경우 사용해줘야한다.

   -w : "Working directory", Container의 내가 현재 있는 곳(pwd)을 고정시킨다.

   --restart=always : Container가 죽으면 자동으로 재시작시킨다. Host 리부팅 또는 Docker 서비스 재시작 등을 대비.
   ```


3. docker exec -it <컨테이너이름> <사용할 터미널>
   - "interactive", "tty". 특정 터미널 프로그램을 통해 컨테이너 내부로 들어가겠다. (콘솔 접속 느낌)
   - 최초 접속 경로는 ***container image를 만든 사람이 설정한 Working Directory로 접속***된다.
   - ctrl p, q : "조용히" 나온다. foreground에 돌고 있는 서비스를 건들지 않고 Host로 나온다.
   - ctrl d : "그냥" 나온다. 만약 컨테이너 내부에 foreground로 동작하는 프로세스가 없다면 컨테이너 자체가 꺼진다.
  
     
4. docker rm
   - 컨테이너 삭제. -f를 붙여서 강제로 삭제.
     > docker rm -f $(docker ps -qa) : 모든 컨테이너를 삭제하는 명령어.

     
5. docker image list
   - Host 저장 container image 리스트를 조회한다. Host에 저장되지 않은게 있으면 docker.io(Hub)로 접속해서 다운해온다.
  
     
6. docker inspect <컨테이너 이름>
   - 컨테이너의 정보를 디테일하게 조회한다. Pipe grep이랑 같이 써서 조회하는게 필요한 것만 확인할 수 있다.
  
     
7. docker cp <Host에 저장된 파일 경로> <컨테이너 이름>:<컨테이너 내부 타겟 경로>
   - 파일을 복사해온다. 순서 바꾸면 Source가 스위치된다.
  
     
8. docker network list
   - 컨테이너에 설정된 네트워크 정보를 불러온다. 총 3개의 driver(형태)가 존재한다.
   - host : publish 불가능한 상태 / null : 네트워크 없음(단일 컨테이너) / bridge : publish 가능(VMware NAT와 유사)
