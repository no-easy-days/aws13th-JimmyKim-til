## 1. docker compose 기본
- "docker run" 방식은 오로지 1개의 컨테이너만 띄울 수 있다.
- "docker compose" 방식은 ***한번에 여러 개의 컨테이너를 동시에*** 띄울 수 있다.
- "docker compose" 에서는 컨테이너를 ***"services"로 지칭***하며, compose.yaml 파일을 통해 구성한다.
  ```
  < docker-compose.yml = compose.yaml 파일의 구조>

  services:
  # 내가 띄울 다양한 종류의 컨테이너"들"
    <컨테이너이름>:
        image: <레지스트리>/<레포지토리>:<태그>
        # 불러올 Container image.
        ※ Dockerfile을 활용해서 Build를 할 경우에는 지정할 image 이름으로 변경됨.
  
        ports:
         - '<배포할 포트>':'<실제 서비스가 사용하는 포트>'
        # Publish 옵션. -p와 동일하다.

        networks:
         - <네트워크 이름>
        # 붙여줄 네트워크 지정. Top-Level에 networks로 생성해줘도 되고 원래 네트워크 불러와도 된다.
        ※ 만약 새로운 네트워크를 지정해줬을 때, external: true 설정을 걸면 IP 주소를 새롭게 받아온다.
        ※ 네트워크 분리는 “컨테이너 간 접근 제어”를 위한 것이다!
           예시 : WEB은 WAS, DB랑 통신되어야하고 WAS는 DB랑'만' 통신되어야하는 그런 상황일 때 사용

        volumes:
         - <호스트경로>:<마운트할 컨테이너의 경로>
        # 호스트 ~ 컨테이너 마운트. -v와 동일한데, Dockerfile이랑 다르게 Host의 어느 경로든 가능!
  
        environmnet:
         - ENV=test
        # 환경변수 옵션. -e와 동일하다.

        depends_on:
         - <먼저 띄워져야하는 컨테이너 이름>
        # 의존성 설정. 먼저 띄워야지 구동할 수 있는 경우(DB!) 사용한다.

        command:
         - "<명령어>"
        # 컨테이너 running되고 실행할 명령. CMD랑 동일하다. 
  ```


- docker compose up -d : 백그라운드로 컨테이너들(서비스들) 구동 시작. 당연히 위의 yaml 파일 참조함.
- docker compose down : compose로 띄운 컨테이너들 중지 & 삭제
- docker compose ls : compose로 띄운 컨테이너들 상태 조회
- docker compose scaling --scale <구동중인서버 이름>=<컨테이너 갯수> : 컨테이너 갯수 늘리거나 줄이거나.


***※ docker compose를 사용하면 굳이 --link 옵션 없어도 "서비스명" 으로도 찾아갈 수 있다.***


## 2. image build도 하고 서비스도 구성할 경우
- Dockerfile을 준비하고, compose.yaml 파일에 옵션 추가하면 된다.
- build는 한번만 수행한다. 그게 싫으면 → ***"docker compose up -d --build"***
  ```
  services:
    <컨테이너이름>:
        build: .
        # compose.yaml 파일이 있는 경로(.)의 Dockerfile을 Build 하겠다.
          context: <Dockerfile 저장 경로>
          dockerfile: <Dockerfile 이름>
          # Dockerfile이 다른 곳에 저장된 경우 / 이름이 다를 경우 사용

        image: <레지스트리>/<레포지토리>:<태그>
        # 빌드 후 저장할 이미지 이름. (위에 빌드 안했을 경우랑 다른걸 기억하자!)
  ```


## 3. docker swarm
- docker compose : 단일 Host에 다수의 Container를 구현한다.
- docker swarm : 다수의 Host에 다수의 Container를 구현한다. ※ ***다수의 Host를 "Cluster"*** 로 칭한다. 
- 컨테이너 오케스트레이션 툴이며, K8s의 기능과 같다고 생각하면 된다. (다만, K8s는 더 어렵지만 부가 기능이 더 많음)
  > Load-balancing & Auto-Scaling, Health-check 등의 다수의 호스트로 구성된 클러스터에서 다수의 컨테이너를 관리하는 기술.


### ※ manager node(Control Plane)가 worker node(data Plane)을 관리한다! (K8s에서 더 자세히 다룰 예정)


- docker swarm이 구동되는 순간, ***"Overlay-Network"*** 가 구성된다. 이 덕분에 다른 컨테이너들 끼리 통신 가능해짐.
  > docker network ls 입력하면 DRIVER에서 overlay로 할당된 것을 볼 수 있음
- 이로 인해 어느 호스트로 접근하던지 동일한 Overlay-Network로 접속하게 된다.


1) 대상이 되는 Host들을 Master / Worker Node로 분류하고, 할당한다.
   > 이 행위에 대해 Amazon ECS에서는 Cluster 형태로 자동 생성&할당 해준다고 생각하면 된다.
   ```
   docker swarm init
   # Manager Node로 선택된 Host에서 수행하면, Worker Node Host를 위한 접속 Command Line & token을 발급해준다.

   docker service create --replicas <컨테이너 복제본 갯수> <docker run과 동일한 옵션 파라미터> <이미지>
   # --replicas : 컨테이너를 여러개 띄우겠다.
   ※ docker swarm을 활용하면 자체적으로 Load-Balancing 기능이 활성화 된다.

   docker service ls
   # 구동중인 서비스 목록의 개략적인 정보를 조회.
   ※ REPLICAS 컬럼을 조회하면 <Current State ~ 현재상태> / <Desired State ~ 원하는 상태> 를 확인 가능하다. 

   docker service ps <특정 서비스 이름>
   # 특정 서비스 상세 조회, 여기서 Node가 어떻게 할당되었는지 확인할 수 있다.
   ```


2) swarm을 위한 설정 파일 생성
   - compose.yaml 파일이 필요하다. 기존 docker compose와 흡사하지만 조금 다름.
     ```
     services:
       <컨테이너 이름>:
         deploy:
           replicas:
             placement:
               constraints: [node.role == manager]

     ※ Manager 인지 Worker인지 constraints에서 할당할 수 있다.
        image 정보는 Worker가 당겨온다. Manager가 분배해주는 느낌으로 오해하면 안된다.
     ```

3) swarm 활성화
   ```
   docker stack deploy -c <compose파일경로> <Stack이름>
   # 생성한 docker swarm을 stack이라는 명령어로 일종의 "클러스터 활성화"
   
   docker stack rm <Stack이름>
   # stack 삭제
   ```


### ※ Docker out of Docker (DooD)
- 컨테이너 내부에서 명령을 내리는데 마치 호스트에서 명령을 내린 것과 동일한 효과를 일으키는 기법.
  > 원래 App들은 컨테이너에 그대로 있지만, 마치 Host에 존재하는 것 처럼 동작함.


- Host ~ Container 간 socket 통신이 대표적인 예시이다. ***"docker.sock"***
