## 1. docker compose 기본
- "docker run" 방식은 오로지 1개의 컨테이너만 띄울 수 있다.
- "docker compose" 방식은 ***한번에 여러 개의 컨테이너를 동시에*** 띄울 수 있다.
- "docker compose" 에서는 컨테이너를 ***"services"로 지칭***하며, compose.yaml 파일을 통해 구성한다.
  ```
  < docker-compose.yml = compose.yaml 파일의 구조>

  services:
  # 내가 띄울 다양한 종류의 컨테이너"들"
    <컨테이너이름>:
        image: <레지스트리>/<레포지토리>:<태그>
        # 불러올 Container image.
        ※ Dockerfile을 활용해서 Build를 할 경우에는 지정할 image 이름으로 변경됨.
  
        ports:
         - '<배포할 포트>':'<실제 서비스가 사용하는 포트>'
        # Publish 옵션. -p와 동일하다.

        networks:
         - <네트워크 이름>
        # 붙여줄 네트워크 지정. Top-Level에 networks로 생성해줘도 되고 원래 네트워크 불러와도 된다.
        ※ 만약 새로운 네트워크를 지정해줬을 때, external: true 설정을 걸면 IP 주소를 새롭게 받아온다.
        ※ 네트워크 분리는 “컨테이너 간 접근 제어”를 위한 것이다!
           예시 : WEB은 WAS, DB랑 통신되어야하고 WAS는 DB랑'만' 통신되어야하는 그런 상황일 때 사용

        volumes:
         - <호스트경로>:<마운트할 컨테이너의 경로>
        # 호스트 ~ 컨테이너 마운트. -v와 동일한데, Dockerfile이랑 다르게 Host의 어느 경로든 가능!
  
        environmnet:
         - ENV=test
        # 환경변수 옵션. -e와 동일하다.

        depends_on:
         - <먼저 띄워져야하는 컨테이너 이름>
        # 의존성 설정. 먼저 띄워야지 구동할 수 있는 경우(DB!) 사용한다.

        command:
         - "<명령어>"
        # 컨테이너 running되고 실행할 명령. CMD랑 동일하다. 

  ```


- docker compose up -d : 백그라운드로 컨테이너들(서비스들) 구동 시작. 당연히 위의 yaml 파일 참조함.
- docker compose down : compose로 띄운 컨테이너들 중지 & 삭제
- docker compose ls : compose로 띄운 컨테이너들 상태 조회
- docker compose scaling --scale <구동중인서버 이름>=<컨테이너 갯수> : 컨테이너 갯수 늘리거나 줄이거나.


***※ docker compose를 사용하면 굳이 --link 옵션 없어도 "서비스명" 으로도 찾아갈 수 있다.***


## 2. image build도 하고 서비스도 구성할 경우
- Dockerfile을 준비하고, compose.yaml 파일에 옵션 추가하면 된다.
- build는 한번만 수행한다. 그게 싫으면 → ***"docker compose up -d --build"***
  ```
  services:
    <컨테이너이름>:
        build: .
        # compose.yaml 파일이 있는 경로(.)의 Dockerfile을 Build 하겠다.
          context: <Dockerfile 저장 경로>
          dockerfile: <Dockerfile 이름>
          # Dockerfile이 다른 곳에 저장된 경우 / 이름이 다를 경우 사용

        image: <레지스트리>/<레포지토리>:<태그>
        # 빌드 후 저장할 이미지 이름. (위에 빌드 안했을 경우랑 다른걸 기억하자!)
  ```
